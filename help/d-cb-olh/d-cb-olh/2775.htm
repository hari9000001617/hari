Derived Attribute Table Form

The Derived Attribute Table form is used to populate a predefined lookup table with the required lookup (index) and result values.

Table of values containing index values and associated result values. The table is used to look-up the result value based on an index value that is calculated using an associated index expression.

Context: Convergent Billing

See also Index Expression, Index Value, and Result Value.

Each row in the lookup table contains at least one result value for each index value.

To expand the:

  • Range of index values covered by the lookup table, add further rows of data.
  • Number of result columns, add fields to the entity validation definition used for the lookup table.

A Search Criteria group is displayed on the form if 'search' entity validation (using the Search Validation field) is defined on the Derived Attribute Definition form.

The Search Criteria section is displayed if entity validation is specified in the Search Validation field when defining the derived attribute array. Search criteria can be used to filter the table rows displayed in the form.

Match Case and Wildcard check boxes are only available for string search fields. If the Match Case check box is selected, a case-sensitive search is executed. If the Wildcard check box is selected, the wildcard character % can be used in the string to represent zero or more unknown characters. For example, ‘%value’ returns 1value, onevalue, and twovalue.

When a filter is applied, another filter cannot be applied (further searches are conducted on the unfiltered array).

Reordering records while a filter is applied is not possible. The Up Arrow and Down Arrow buttons at the bottom of the Derived Attribute Definition form are unavailable while a filter is applied.

If the Where clause is defined in the search entity validation, filtering fields are ignored. If the Go To functionality is used, the specified row is returned regardless of filtering values. Pre-commit and post-commit expressions are also ignored for search entity validation.

Rows can be inserted, updated, and deleted when filtering is applied. Row insertion behaviour is configurable using the SYSTEM_CLIENT configuration items. The APPEND_ON_DA_TABLE_INSERT and DUPLICATE_ON_DA_TABLE_INSERT attributes in the SYSTEM_CLIENT configuration item (see the System Configuration Guide for Singleview Convergent Billing) control how rows are added to the form.

The default behaviour of the APPEND_ON_DA_TABLE_INSERT attribute is to append rows at the end of the derived attribute table. If set to BOOLEAN_NM_FALSE, a row is inserted before the selected row. The sequence numbers of the selected row, and subsequent rows, are incremented by 1.

The default behaviour of the DUPLICATE_ON_DA_TABLE_INSERT attribute is to duplicate values to the new row. If set to BOOLEAN_NM_FALSE, the new row is blank.

The buttons at the bottom of the form are used to navigate through the rows in the derived attribute table.

The following three buttons ('Go To' functionality) at the bottom of the form are used to navigate through the rows in the derived attribute table:

Go to row number

Go to first record

Go to last record

The Go to row number button is used to go directly to a row by entering a row number in the text box to the left of the button and clicking the button. The 100 rows before and after the entered row number are also returned.

Note:

When the Go to row number button is clicked, the form is cleared, and the required records are retrieved from the database. Therefore, any filtering or sorting used in the grid is cleared when the button is clicked.

The Go to first record button is used to go to the first record displayed in the grid (filtering and sorting is not cleared).

The Go to last record button is used to go to the last record displayed in the grid (filtering and sorting is not cleared).

The Go to first record and Go to last record buttons display the first and last rows, respectively, displayed in the grid, not necessarily the first or last records in the derived attribute definition table. For performance reasons, the Convergent Billing client only retrieves records from the database as they are required; therefore, the derived attribute definition table is not always retrieved when it is initially displayed in the client. For example, for a table with 1000 records, only the first 32 may be initially retrieved and displayed. When the user scrolls past row 32, the next set of records are retrieved. Therefore, clicking the Go to last record button when the table is first displayed may take the user only to row 32, not the last row in the derived attribute definition table.

If it is necessary to delete multiple rows from the table:

  1. Press the Shift or Ctrl key while selecting the rows.
  2. Press Ctrl+Delete, or select the Delete button to delete the rows.

Associated Actions

Defining a Derived Attribute

To define a derived attribute:

  1. Select the Maintenance Tariffs Derived Attributes Definition option. The Derived Attribute Search form is displayed.
  2. Click the New... button. The Derived Attribute Definition form is displayed in Insert mode.
  3. Complete the data entry fields as required.

    Evaluation Type: Conditional: Single Value Returned

    Evaluation Type: Conditional: Multiple Values Returned

    Evaluation Type: List

    Evaluation Type: Simple Expression

    Evaluation Type: Table Lookup by Best Match

    Evaluation Type: Table Lookup by Multiple (1-10) Keys

    Evaluation Type: Table Lookup by Searching

    To define a derived attribute having an evaluation type of Table Lookup by Searching:

    1. View the Derived Attribute Definition form.
    2. Complete the data entry fields on the Details tab. Ensure that the Evaluation Type field is set to Table Lookup by Searching.
    3. Select the appropriate storage context.
    4. Complete the appropriate data entry fields.
    5. Select the Lookup Table tab. Select the appropriate entity validation definition in the Validation field, ensuring that the definition is defined with at least one index and one result column.
    6. If a user-specified wildcard is required, enter the wildcard value in the Wildcard String field on the Lookup Table tab. Otherwise, leave the field blank and undefined values are treated as wildcards.
    7. If required, enter the result column number and maximum number of rows.
    8. Select the Indexes tab. Enter the appropriate index expression.
    9. Click the Save button on the toolbar. If a storage context of Global or Global-Shared is used, continue with the procedure.
    10. Click the Table… button on the Lookup Table tab. The Derived Attribute Table form is displayed with the defined index and result columns.
    11. Populate the table with the required lookup (index) and result values.
      • If a user-specified wildcard is set in the Wildcard String field, ensure that the wildcards entered in the index columns match the value in the field.
      • If Wildcard String field is not set, any undefined index columns are treated as wildcards.
      • A row can be included in the derived attribute table with all index columns specified as undefined wildcards. This is the default row returned if no matches are found.
    12. Click the Save button on the toolbar.

    Evaluation Type: Table Lookup by Single Range.

  4. Click the Save button on the toolbar.

Defining a Slave Derived Attribute

To define a slave derived attribute:

  1. View the Derived Attribute Definition form.
  2. In the Master Derived Attribute field, select the name of the derived attribute to act as the 'master' for this 'slave'. Details associated with the specified master derived attribute are displayed on the current form.

    Note:

    The Master Derived Attribute field is enabled only if one of the table lookup evaluation types is specified.

  3. Change the value of the Result Column Number field to a number other than one. The number one is used to identify the result column associated with the master derived attribute.
  4. Complete the data entry fields as required.

    Evaluation Type: Simple Expression

    Evaluation Type: Conditional: Single Value Returned

    Evaluation Type: Conditional: Multiple Values Returned

    Evaluation Type: List

    Evaluation Type: Table Lookup by Best Match

    Evaluation Type: Table Lookup by Multiple Keys

    Evaluation Type: Table Lookup by Searching

    Evaluation Type: Table Lookup by Single Range.

  5. Click the Save button on the toolbar.

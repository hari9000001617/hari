Charge Category Examples

The following table lists examples of how charge categories can be defined, overridden, and used.

 

Example 1

Basic Charge Category

Requirement

Costs and benefits are to be posted to the primary account of the customer node. GL codes are not used.

Solution

Define the base default charge category without specifying either an account number or a GL code, but specifying the same account type as the primary account. The account number of the customer node's primary account is selected automatically when the charge category instance is defined.

Example 2

Assigning Charges to Another Customer Node

Requirement

Another department (with its own customer node) agrees to pay the charges for a service defined at a customer node.

Solution

Amend the charge category instance for the service, by replacing the account number with the primary account number of the other customer node.

Example 3

Splitting a Charge between Customer Nodes

Requirement

Two customer nodes share a service defined at one of the nodes. The other customer node agrees to pay 30% of the charges for the service.

Solution

  1. Define a duplicate of the charge category instance, with the account number set to the primary account of the other node, and the proportion set to 0.3 (the proportion this node will pay).
  2. Amend the existing charge category instance, by changing the proportion to 0.7 (the proportion paid by this customer node).

    Note:

    When charges are split in this way, the sum of all the proportions defined must be 1.

Example 4

Using a different Account Type

Requirement

The company awards loyalty points based on usage. The number of points accumulated is to be recorded in an account of type Loyalty, defined at the customer node.

Solution

Define the base charge category with the account number empty, and the account type set to Loyalty. When the charge category instance is defined, the number of the Loyalty account at the customer node is entered automatically.

Note:

  • The appropriate account type must be defined before it can be used in the default charge category.
  • The service instance cannot be activated if the customer node does not have an account of the appropriate type.

Example 5

Tracking Payments Owing to a Supplier

Requirement

Another service provider provides a service being sold to customers, and it is necessary to track the payments due to this provider.

Solution

  1. Create a customer node for the supplier. Payments due to this supplier can then be accumulated in the primary account.
  2. Define a tariff for the charges to be paid by customers in the usual way and include it in the product definition. A charge category instance at each customer node posts charges to the appropriate account.
  3. Define another tariff to calculate the payment due to the provider, and include it in the product definition.
  4. Set the account number of its default charge category to point to the primary account of the provider's customer node. The charge category instance for this tariff then posts charges to the supplier's account.

Example 6

Using the ‘From’ Account

Requirement

A tax based on usage is payable by the customer. The tax due to the tax authority, for whom the company is acting as a collecting agent, must be recorded.

Solution

  1. Create a customer node for the tax authority.
  2. Define a tariff for the tax payable, and add it to the product definition.
  3. Define the base default charge category with the 'to' account number empty (for automatic assignment of the customer's account number when the charge category instance is created), and the 'from' account number set to the primary account number of the tax authority's customer node.

    Note:

    The charge type of the base default charge category should be set to 'Charge Transfer'.

Example 7

Charge Category for a Global Tariff

Requirement

The company wants to keep a record of all calls to a specified destination switch.

Solution

Define:

  1. A global tariff with context Normalised Event that calculates the desired quantity. Set the eligibility for this tariff to identify the calls to be recorded.
  2. The base default charge category for this tariff to point to the account in which to store the call information.

    Note:

    The solution is one of the few times when an account number is defined in the base default charge category, because a global tariff uses this default directly, rather than an instance of it. If the base default charge category does not include an account number, the tariff does not work.

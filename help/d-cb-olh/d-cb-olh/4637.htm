Transaction Engine (TRE)

Convergent Billing has a three-tier client/server architecture:

  • The first tier consists of clients, remote hosts, and external applications, using a workstation or remote functions to enter, inspect, modify data, and initiate server processes.
  • The second tier consists of:
    • Convergent Billing's expression-driven configuration and the Transaction Engine (TRE). Configuration is used to reflect the customer's individual business logic. The second tier manages transactions initiated from the first tier and returns results to the first tier.
    • Convergent Billing processes and the TRE services that clients can use for core business functions. In addition, an Application Programming Interface (API) is available to provide other applications with access to TRE functionality.
  • The third tier is the Oracle database, in which business and administrative data is stored.

The TRE is the second tier of the Convergent Billing architecture and is a configured installation of Tuxedo middleware. The TRE has the primary function of ensuring data exchange between clients, server processes, and the database. EPM built-in, callback and user-defined functions supplied with Convergent Billing are called from EPM expressions and functions defined by the user. The user may also define functions within a Tuxedo service, provided the appropriate service is configured.

The TRE consists of several server processes. The processes are divided into two categories: Tuxedo processes, and Convergent Billing processes. The following lists these processes:

  • Tuxedo processes:
    • Bulletin Board (BBL)
    • Transaction Management Server (ORATMS: two instances)
    • Workstation Listener (WSL) and Handler (WSH)
  • Convergent Billing processes
    • TRE named servers

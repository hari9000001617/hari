Subtotal Definition

Subtotals can have an application environment of either Rating or Billing and a context must be specified. Customer contexts can also be created using keyed subtotals. Terms for rating subtotals can only have an evaluation context of Normalised Event or Charge.

Subtotal with a data type of array or hash. A value is calculated for each value of the key in the aggregation set, so that multiple values are returned for a single subtotal. Keyed subtotals are often used in conjunction with keyed tariffs and keyed derived attributes, to provide tiered discounts, or aggregation for related products.

Context: Convergent Billing

See also Aggregation, Subtotal, Keyed Tariff, and Subtotal Term.

When defining a subtotal, its status, which can be set to Active (the subtotal is in use), or Trial (the subtotal is being tested or undergoing trials) can be specified. Subtotals are available to the billing engine for a billing run, regardless of the status.

By default, the value of a persistent subtotal is stored and used in the future (for example, in conjunction with user-defined functions). For example, permanent (or persistent) storage of subtotal values allows the charges for the current billing period to be compared with the last billing period or with the same billing period for the previous year.

A subtotal is defined by one or more subtotal terms. For more information about subtotal terms, refer to the Customer Care and Product Configuration Guide for Singleview Convergent Billing.

Note:

The subtotal name must be unique for the date range of the subtotal when compared to all other variables (for example, functions, derived attributes, subtotals, tariffs, and direct variables). The rules for naming are:

  • For a scalar data type, the name and data type combination must be unique when compared to other variables.
  • For a hash type, the name must be unique when compared to other hash type variables.
  • For an array type, the name must be unique when compared to other array type variables.

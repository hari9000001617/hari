Table Partitioning and Scalability

Partitioning handles the problem of supporting very large tables and indexes by allowing them to be split up. Partitions increase efficiency by splitting up large tables or indexes into smaller and more manageable sections.

Table partitions are defined during implementation as part of the database build process. When planning table partitions, future growth should be considered. Ideally, the process of creating table partitions should be a one-off build process.

Partitioning is used by the following two types of scalability to improve performance:

  • Vertical

    Used for a single instance, and involves partitioning a specified subset of tables by date.

  • Horizontal

    Used for multiple instances, and partitions all tables containing customer instance data on the primary identifier for the table.

For information about the architecture of Convergent Billing for vertical and horizontal scalability, refer to the Architecture Overview for Singleview.

Convergent Billing supports the use of the Oracle partitioning option for the management of data in the Convergent Billing database. The tables that can be partitioned are defined in the TablePartition derived attribute table.

  • DOWNLOAD_AUDIT

UPLOAD_AUDIT.

Partitioning is recommended to:

  • Improve rating performance.
  • Improve billing performance.
  • Simplify archiving of old event, charge, and subtotal data.
  • Simplify database administration tasks associated with tables.

Convergent Billing provides the capability to define and manage table partitions using the Table Partition Definition form. Local partitioning indexes are used for each table. Schedule types are supplied to automate most activities associated with the creation and maintenance of partitions on the tables.

For more information about:

  • Table partitioning, refer to the System Configuration Guide for Singleview Convergent Billing.
  • Oracle partitioning features, refer to Oracle documentation (for example, the Oracle Concepts Guide).

Bill Runs

In Convergent Billing, bill run types are used to configure what tasks run together for a specific bill run.

Bill run types offer the option of running a range of billing operations, from adjusting rental events to un-applying invoices. Generating rental events (RGP), generating invoices (BGP) and invoice images (IGP), and outputting those results (GOP) are all possible tasks for a bill run.

For example, a standard bill run schedule could be configured to perform an immediate bill run consisting of the BGP, IGP, and to apply invoices, but not to run the RGP or the GOP. If required, the apply invoices and revoke invoices processes can also be run as individual schedules. For example, if a daily cycle bill run fails and it is determined that one customer caused the failure, the whole bill run does not have to be revoked and rerun; only the customer who caused the failure needs to be revoked and rerun.

Using bill runs, it is possible to generate invoices, which are sent out to the customer for payment, and temporary statements, which can be used for quality assurance purposes or for customer information. Invoices can also be generated for designated interim rental events (tariffs), so a customer can be invoiced for a specific product or service (using the Interim Bill Run schedule type).

Processing multiple customer nodes speeds up the bill run process, and controls the quality of the process. For example, the user can review initial invoices before the majority of customer nodes have been processed; if the first invoices contain errors, the bill run can be stopped.

Statistical information for each billing operation is recorded, including the success or failure of the operation and its duration. For bill runs processing multiple customers, the statistical information is recorded for each operation and summarised for the overall bill run.

Billing personnel also have complete control over how many errors can occur before the bill run is aborted. If errors occur, it is necessary only to revoke the billing operations on the customer nodes that failed.

Attribute Types

An attribute type controls the representation of an entity's attributes.

For users of Singleview Convergent Billing, an attribute type defines the format, characteristics, and validation rules for a form field.

For external systems, attribute types control representation of an entity's data model. In this context, attribute types can define:

  • Fields that uniquely identify an entity instance
  • Labels that can be used to access fields of an entity
  • A field's value as a reference to another Convergent Billing entity
  • A non-stored attribute that is derived from an entity's stored fields.

Attribute types are defined independently of the data entry fields to which they relate, so the same attribute type to be used for multiple fields.

Attribute type configuration can be used to add security to a field using any of the following features:

  • Adding encryption
  • Adding data masking
  • Restricting view and update access for specific security groups
  • Enabling or disabling access for specific security groups.

In conjunction with reference types, attribute types allow generic Convergent Billing fields to be customised using entity validation.

Note:

Customising fields includes adding new fields. The customer care forms include a General tab, which contains invisible general fields that can be customised to become visible.

To speed up data entry and reduce the scope and level of validation required, a reference type can be specified in an attribute type so the user can select one of a number of pre-defined values rather than having to determine, and then enter, an appropriate value in the correct format.

For more information on attribute types, refer to the Configuration Fundamentals Guide for Singleview Convergent Billing.

Derived Non-Stored Attributes

Derived non-stored attributes are derived from an entity's stored attributes. They can be added to an entity validation without a corresponding validation table column. To external systems using the Entity Validation Server, derived non-stored attributes appear the same as stored attributes. Derived non-stored attributes can tailor an entity's data model for integration with external systems.

Derived non-stored attributes are defined on the Integration tab of the Attribute Type Definition form. The Derived? check box enables the details of the derivation to be configured.

Table 1 lists three types of derived non-stored attributes.

 

Table 1 Types of Derived Non-stored Attributes

Type

Description

Specified on Input

The Specified on Input? check box is selected.

External systems can insert or update the attribute value, but are not able fetch it or generate a view that contains it.

How the supplied value is broken down into values for stored attributes is defined in the validation expressions of the entity's entity validation.

Specified on Output

The Specified on Output? check box is selected.

External systems can fetch the attribute value or generate a view that contains it, but are not able to insert or update it.

The derivation of the attribute value from stored attributes for a fetch is defined in the Output EPM Expression field.

The derivation of the value for a generated view is specified in the Output SQL Expression field.

Specified on Input and Output

The Specified on Input? and Specified on Output check boxes are selected.

External systems can fetch, insert, and update the attribute's value. They can also generate a view that contains the attribute's value.

Derivation of the output value is defined as per Specified on Output.

Breaking down the input value is defined as per Specified on Input.

 

Derived non-stored attributes can be referenced in EPM expressions using a direct variable. The name of the direct variable is configured in the Variable Name field.

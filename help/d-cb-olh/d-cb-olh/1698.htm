Direct Variables

A direct variable is a variable mapped to data held in a database table or to a column in a database view. The value of the variable is determined by the context in which the variable is used. For example, each field of a normalised event record is mapped to a direct variable.

During normalisation of event data, each data field of a normalised event record is stored as a direct variable. The variables into which the field values are mapped are defined specifically for the type of record being normalised. Product Administrators do not need to know what these mappings are, but to know what direct variables are available.

For example, to define a usage tariff, call data information (for example, calling party ID, called party ID, and call duration) needs to be referenced. Data information is stored in direct variables (for example, APartyId$, BPartyId$, and Duration#) when a call data record is normalised and is therefore available directly.

Direct variables therefore allow a field in a database record to be referenced, and the referenced value used in expressions. Direct variables can also be used as the basis for other variable types.

If required, additional direct variables can be defined, which requires knowledge of SQL and the names and structure of the Convergent Billing database tables.

Like other variables, direct variables can be referenced in tariffs and subtotals by including the name of the variable.

Note:

Direct variables are always defined with an application environment of Data Entry.

Direct variables created during normalisation should usually cover most requirements. However, if required, a direct variable can be defined. This requires knowledge of the available Convergent Billing database tables and knowledge of SQL.

Specifically, a field/column in a database view needs to be identified by specifying a view name, column name, and column number.

Note:

The direct variable name must be unique for the date range of the direct variable when compared to all other variables (for example, functions, derived attributes, subtotals, tariffs, and direct variables). The rules for naming are:

  • For a scalar data type, the name and data type combination must be unique when compared to other variables.
  • For a hash type, the name must be unique when compared to other hash type variables.
  • For an array type, the name must be unique when compared to other array type variables.

Direct Variables Associated with Attribute Types

Direct variables associated with attribute types are configured on the Integration tab of the Attribute Type Definition form. The Variable Name field defines a direct variable that can be used to reference a derived non-stored attribute value in entity validation expressions. Completing the Variable Name field automatically creates a direct variable with:

  • An application environment of Data Entry
  • A context of Any
  • Associated view names as appropriate for the entity.

Direct variables associated with attribute types can be viewed on the Variable Definition form, but cannot be updated. They have no stored value, so the Column Name field is blank for these variables.

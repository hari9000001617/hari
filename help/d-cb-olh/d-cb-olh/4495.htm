Archiving

The terms 'archiving' and 'un-archiving' imply the exporting or importing of data relating to specific database entities, taken from one or more tables in the database. For example, the entity customer contains information about the individual customer, the customer hierarchy, the people nominated as contacts, the accounts involved, invoice details, and much more.

An entity is made up of the base level data (for example, customer details) plus all dependent data, wherever it is stored in the database.

Archiving is not synonymous with backup and restore, which are generally carried out on one or more complete tables (if not the entire database), and are used to provide a means of recovering from any damage the database may suffer as a result of hardware or software failure.

Archiving is more generally used to remove and archive database entities no longer required; for example, customers who have permanently closed their accounts, people who have moved away, or products no longer available. As data of this kind is archived, it can also be automatically purged from the database.

Another common use for archiving is the transfer of entity data, spanning several tables, from one database to another. For example, a subset of data from an operational database might be archived, then imported into a training database, to provide realistic dummy data for students to work on. In this case, the original data would not be purged after archiving.

The archiving feature can be used to purge unwanted data from the database, such as dummy data entered by students in a training database when their training is complete and the data is no longer required. Purging is done by using the archiving feature with no output file specified and the purge option turned on. Data purged in this way is lost forever, so the technique should only be used on unwanted data.

Caution:

The archiving feature is not designed to archive large volumes of configuration or operational data. For example, the archiving feature should not be used to archive or remove a large set of subscribers from a production system. Not only does archiving or purging large volumes of data take considerable time, but it is likely that system performance is significantly degraded during the archiving period.

In some systems, even archiving the operational data associated with a single customer may not be advisable. For example, a customer with a high volume service (such as a government body) may have millions of entries in the normalised event table. Charge and normalised event tables contain the highest volume of data in the system and the period to archive the customer node and its associated data may be lengthy.

Archiving can be carried out from the client workstation by scheduling a task, or from the UNIX command line. All options available from the command line are also available through the scheduler. In either case, the key parameter that the archiver needs is an Archive type, which specifies all the tables and dependencies that make up the entity being archived. Archive types are provided with the base installation, and cannot be defined or changed by users.

Convergent Billing archiving uses either the Oracle imp and exp utilities, or sqlimport and sqlexport if the archive files are required in ASCII format.

Entity Validation Definition Form

The Entity Validation Definition form is used to enter, view, and update an entity validation definition.

Used to customise fields, add fields to generic forms, validate fields and forms, and add field security. It can also be used to customise an entity's data model using derived non-stored attribute types, attribute display labels, attribute interface labels, and custom key field functions.

Entity validation is used in conjunction with reference types, attribute types, lookup functions and context-sensitive help.

Context: Convergent Billing

See also Attribute Type and Reference Type.

The upper part of the form is used to enter basic details about the entity validation definition, select validation table and a parent table, specify key field functions, and enter validation expressions or post-commit expressions.

Contains variables that can be used in the default expression of an entity validation definition.

Context: Convergent Billing

See also Entity Validation and Validation Table.

Table used within entity validation that contains a group of generic fields, including general fields, which can be customised. Each table represents a specific area such as accounts, addresses, adjustments, and so on.

Context: Convergent Billing

See also Entity Validation and Parent Table.

The lower part of the form is used to associate attribute types with generic Singleview Convergent Billing fields, use the Control Next feature, and enter default expressions.

The default expression in entity validation can execute almost any action; for example, assigning a default to a field, and printing a message.

To define a default for a field, the expression must contain the direct variable for the field referenced in the expression. For example, to define a default of ‘500’ for the General 4 field of the Customer Detail form, the entity validation defining that field would contain the following default expression:

evCustGeneral4$ := ’500’;

where

evCustGeneral4$ is the direct variable for the General 4 field on the Customer Detail form.

Direct variables from both the validation table or parent table can be included in default expressions.

The Control Next feature is used to define intelligent defaulting and to configure fields to automatically change their configuration based on a value selected in another field. For example, Control Next is used to configure the following scenarios if a user selects an option in one field:
  • More information is required based on that selection.
  • An additional field, used to enter further information, is displayed.
  • The characteristics of another field are inappropriate for the option selected, so the characteristics of the field are automatically modified.

Note:

The Control Next feature does not work on grids, except for derived attribute grids where child reference types are used.

Changing the configuration of fields based on the data entered in a separate field is achieved by allowing the attribute type assigned to one field to control the attribute type assigned to another field.

If the Control Next field (in an entity validation definition row) is set to Yes, the attribute type in that row has control over the attribute type in the row directly below it. Depending on how the attribute types are set up, the Control Next feature can have the following effects:

  • If the controlling attribute type is based on a reference type that has a child attribute type associated with one of its reference items, that child attribute type overrides the attribute type in the next row when that reference item is selected.

    The result is that the second field changes its configuration when an option is selected from the first field.

    Note:

    To display a new field on the form when an option is selected, an attribute type defining an invisible (or null) field must be defined. The attribute type defining the blank field is then overridden and the new field is displayed.

  • If the controlling attribute is based on a reference type that has a child reference type associated with one of its reference items, that child reference type overrides any reference type specified in the attribute type in the next row when that reference item is selected.

    The result is that the second field changes its reference list (drop-down list) when an option is selected in the first field.

    Note:

    Child reference types are supported only in derived attribute grids; they are not supported in other grids with entity validation.

    For example, they are supported in the Derived Attribute Table form, Customer Lists, and Service Lists; they are not supported in the Payment Batch form, Adjustment Batch form, Customer Invoice Format form, and Customer Contacts form.

  • If the field being controlled has a default expression defined, that expression is evaluated when the field associated with the first attribute type is modified. The above setup is used to define intelligent defaulting (automatically defaulting the value of a field based on data entered in a separate field).

Controls the representation of an entity's attributes.

For users of Singleview Convergent Billing, an attribute type defines the format, characteristics, and validation rules for a form field.

Attribute types are the major components for entity validation. Based on a reference type, they also define the representation of an entity's attributes externally, allowing the entity's data model to be tailored for integration with external systems. They can also define additional non-stored attributes that are derived from an entity's stored attributes.

Attribute types are defined independently of the data entry fields to which they relate, which allows the same attribute type to be used for multiple fields.

Context: Convergent Billing, Process Engine

See also Derived Attribute, Derived Non-Stored Attribute, and Reference Type.

Associated Actions

Viewing an Entity Validation Definition

To view an entity validation definition:

  1. Select the Maintenance Data Definition Entity Validation option. The Entity Validation Search form is displayed.
  2. Enter or select the appropriate search criteria and search method.
  3. Click the Search button. A list of entity validation definitions matching the specified criteria is displayed.
  4. Select the appropriate entity validation definition and click the Detail… button, or double-click the appropriate entity validation definition. The Entity Validation Definition form is displayed in View Only mode.

Defining an Entity Validation Definition

To define an entity validation definition:

  1. Select the Maintenance Data Definition Entity Validation option. The Entity Validation Search form is displayed.
  2. Click the New… button. The Entity Validation Definition form is displayed in Insert mode.
  3. Complete the Entity Name and Description fields.
  4. Select a validation table.
  5. Enter key field functions if required.
  6. Enter validation expressions for Row Validation and Pre-Commit Validation, as required.
  7. Select a field from the Column Name field, then the attribute type to be associated with the field from the Attribute Type field.
  8. Set the Control Next field to Yes if required.
  9. Enter a default expression if required.
  10. To associate an attribute type to another field, position the cursor in the grid section of the form and press the down arrow. A new row is inserted in the grid. Repeat steps 7 to 9.
  11. Enter in-transaction expressions in the In-Transaction Expressions tab, if required.
  12. Enter post-commit expressions in the Post-Commit Expressions tab, if required.
  13. Click the Save button on the toolbar.

Updating an Entity Validation Definition

Caution:

Modifying an entity validation definition may result in corruption of data and forms that use the definition.

To update an entity validation definition:

  1. View the appropriate entity validation definition.
  2. Click the Update button on the toolbar. The Entity Validation Definition form is displayed in Update mode.
  3. Update the entity validation definition as required.
  4. Select the Post-Commit Expressions tab to check if a legacy post-commit expression exists.
  5. If a legacy post-commit expression exists, choose either the Move to In-Transaction List button or Move to Post-Commit List button.
  6. Click the Save button on the toolbar.
  7. If the entity validation definition is used by another element, a Confirm window is displayed. Click the Yes button.

Note:

If the entity validation is being used for a derived attribute and the number of columns or the data types of the columns have changed, the index expression rows on the Indexes tab of the Derived Attribute Definition form must be updated.

Need more information on entity validation?

Deleting an Entity Validation Definition

Caution:

Deleting an entity validation definition may result in corruption of data and/or forms that use the definition.

To delete an entity validation definition:

  1. View the appropriate entity validation definition.
  2. Click the Delete button on the toolbar.
  3. When prompted, click the Yes button to confirm that the entity validation definition is to be deleted.

Security Groups

Security groups are used to control access to the data within the Singleview database. Data belonging to a specific security group can only be seen by users who belong to the same security group. The ability to modify or delete this data is further controlled by access level settings.

The primary security group is associated with operational data created associated with a profile. The configuration security group is associated with configuration data created and modified by users associated with this profile. It defaults to the Primary Security Group.

Data attached to the either security group can only be seen by users who belong to the same security group, if the role defined in their profile allows access to the appropriate menu items and forms.

Note:

Users can belong to more than one security group.

The access privileges for a user belonging to the correct security group is determined by the access levels set in the security profile. Users with sufficient access privileges can change the security group to which data is attached.

When the security group of data is changed, any encrypted data is automatically re-encrypted to ensure that all data within the security group is encrypted with the correct encryption key.

When the security group of an entity with child dependencies is changed, the data associated with the dependent child entities must be re-encrypted manually. To re-encrypt data manually, use the biCustomerNodeReencrypt&() and biDerivedAttributeArrayReencrypt&() functions (see the Configurer's Reference Guide for Singleview Convergent Billing (Vol. 1)).

Form Roles

Form roles provide users with access to specific forms and menu items, and control the updating privileges required when working with each form.

A form role determines which forms can be accessed by a user, access level settings for each form, and whether the menu item for a form is greyed-out.

Function Roles

Function roles specify which functions a user, with the appropriate security settings, can execute using the TRE.

Function roles also define function aliasing; for example, if a user calls one function, the TRE executes the function that is defined as the alias in the function role for that user. TRE function aliasing reduces the configuration costs by making it easier to implement the customer's business rules.

Work Groups

In Convergent Billing, a user or group of users responsible for progressing and resolving queries is called a work group. A work group is similar to a security group in that queries are attached to a work group in the same way that data is attached to a security group. To work with a query, the user profile must assign the user to the work group that created and progressed the query.

Work groups can often be based on workflow. When planning work groups, map the workflow of the various groups. The map provides the information required for determining the work groups that need to be defined and to whom they need to be assigned.

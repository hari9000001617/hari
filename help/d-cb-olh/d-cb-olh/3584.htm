Lists

A list is a defined set of values with a common purpose. For example, a list can contain the numbers frequently called by a specific customer, a list of areas serviced by a specific substation, or a list of the area codes that form the central business district of a specific city.

As in the case of service types, it is inefficient and often impractical to hard-code information into Convergent Billing. For this reason, a derived attribute can be defined, with the type of information held in the associated list. Like equipment types, the values in the derived attribute are not defined during product definition but entered in response to information supplied by the customer.

When a derived attribute, with an evaluation type of List, is evaluated, the value of a conditional expression is compared to each of the values defined in the derived attribute. The derived attribute returns TRUE (a non-zero value) if the given value is defined in the derived attribute, or returns FALSE (zero) if the given value is not defined in the list.

The most common use for a derived attribute, with an evaluation type of List, is used to group frequently called numbers so that special tariffs can be applied, and customers can nominate up to five B party phone numbers that they call frequently.

The specified numbers (entered by a user on the Lists tab on Customer Detail form) are stored in a derived attribute and whenever that customer makes a call, the B party phone number is compared to the numbers in the derived attribute. If a match is found, the special tariff is applied; if no match is found, the normal call tariff for that customer is applied.

The application environment determines the options available for derived attribute storage and use. Derived attributes processed in the Billing application environment, can be used in Customer, Customer Node, Normalised Event, and Service contexts, and stored in Customer, Customer Node, Facility Group, and Service contexts.

Derived attributes processed in the Rating application environment, can be used in the Service context, and stored in Facility Group and Service contexts. Derived attributes can be either master or slaves. Master and slave derived attributes share the same data, but the slave derived attribute may redefine how the common data is accessed.

A derived attribute can also be used to define a list to look up facilities in a facility group and then specify tariff eligibility criteria. The list indicates whether a specific facility is in a facility group and therefore whether tariffs associated with that facility should be applied.

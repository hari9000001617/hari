INV_INVOICE_OUTPUT_V

The INV_INVOICE_OUTPUT_V view defines invoices required to control their output by the General Output Process (GOP).

Column Definitions

Column Name

Properties

Description

INVOICE_ID

NUMBER (15)

Not null

ID of a customer node invoice.

INVOICE_SEQNR

NUMBER (3)

Not null

Sequence number that, with the INVOICE_ID, identifies an invoice image.

OUTPUT_IMAGE_TYPE_CODE

NUMBER (2)

Not null

Type of this invoice image, used to select a compatible output device.

LAST_MODIFIED

DATE

Not null

Date-time when this row was last modified.

CUSTOMER_INVOICE_STR

VARCHAR (30)

Not null

ID generated for this invoice.

The format depends on the implementation, but is usually derived from INVOICE_ID.

INVOICE_TYPE_ID

NUMBER (8)

Not null

Redundant

Type of this invoice, derived from the account type of the invoice's account.

BILL_RUN_ID

NUMBER (15)

Not null

ID the bill run that generated this invoice.

RUNNING_IND_CODE

NUMBER (1)

Set to 1 (TRUE) while the billing engine is in the process of calculating the value for this invoice or statement.

It can be used to quickly identify incomplete invoices associated with a failed bill run that need to be revoked.

QA_IND_CODE

NUMBER (1)

Set to 1 (TRUE) if this is an invoice or statement generated for a Quality Assurance or Statement-only bill run.

It can be derived from the bill run when the invoice or statement is originally created. If not 1, it must be null.

Records with this set must not be applied to accounts.

SUPPRESS_IND_CODE

NUMBER (1)

Set to 1 (TRUE) if this invoice is marked for suppression.

It is set by invoice type expressions, and is not applicable and cannot be set for statements or Quality Assurance invoices.

If all invoices in a customer hierarchy have this flag set for a bill run, the invoices and statements are immediately revoked, including any rental charges and adjustments run for this customer hierarchy and bill run. It is as though the bill run was not run at all for this hierarchy.

The CUSTOMER_NODE_BILL_RUN table indicates that invoice generation for the bill run was suppressed for this hierarchy.

Revoking of suppressed invoices is performed by the bill run operation.

If only some of the invoices in a customer hierarchy have this flag set for a bill run, it has no effect. However, the flag remains set for those invoices in the hierarchy that qualified for suppression.

ACCOUNT_ID

NUMBER (15)

Not null

ID of the account to which this invoice was applied.

ACCOUNT_NAME

VARCHAR (30)

Not null

Name of this account.

CUSTOMER_ACCOUNT_INVOICE_
STR

VARCHAR (30)

Optional ID generated for this invoice.

If defined, the ID is only unique within the account.

Its use is implementation-dependent.

CUSTOMER_NODE_ID

NUMBER (15)

Not null

Derived

Customer node that received this invoice, derived from the ACCOUNT_ID.

APPLIED_IND_CODE

NUMBER (1)

Set to 1 (TRUE) when this invoice is applied to the account.

The value is set by the apply_invoices server utility as appropriate.

After an invoice is applied it cannot be modified in any way (except for flagging it for reprint). Any account corrections must be made by adjustments displayed on the next invoice run for the customer.

The DVP checks that an entry exists in the ACCOUNT_HISTORY table for each invoice with this flag set.

PRINTED_IND_CODE

NUMBER (1)

Set to 1 (TRUE) when the invoice image has been printed as part of the General Output Process (GOP).

It is set by the GOP.

REPRINT_IND_CODE

NUMBER (1)

Set to 1 (TRUE) if a reprint of this invoice image is required; otherwise, it is null.

EFFECTIVE_DATE

DATE

Not null

Effective date of the bill run that generated this invoice.

ISSUE_DATE

DATE

Not null

Date on which this invoice was issued.

PAYMENT_DUE_DATE

DATE

Not null

Date on which this invoice is due to be paid.

EARLY_PAYMENT_DATE

DATE

Not null

If the customer pays before this date, they receive an additional discount.

The discount will have been incorporated into the INVOICE_TOTAL.

INVOICE_AMOUNT

NUMBER (19,3)

Not null

Total amount of charges associated with this invoice, excluding the balance forward.

Values stored in this column are at the limits of double precision.

BALANCE_FORWARD

NUMBER (19,3)

Not null

Balance on the account that was present before the application of this invoice.

ACCOUNT_BALANCE

NUMBER (19,3)

Not null

Derived

Account balance after this invoice was generated, equal to INVOICE_AMOUNT + BALANCE_FORWARD.

It is enforced by the C_INVOICE check constraint.

CURRENT_DUE

NUMBER (19,3)

Not null

Derived

Current amount left to be paid on this invoice, equal to INVOICE_AMOUNT when an invoice is issued.

It is automatically decreased as payments or adjustments are made.

It is automatically adjusted for payments by a database trigger on the PAYMENT_INVOICE table.

Similarly, it is automatically adjusted for adjustments by a database trigger on the ADJUSTMENT_INVOICE table.

Its value is checked by the DVP.

TOTAL_PAYMENTS

NUMBER (19,3)

Not null

Total payments made on this account between the issuing of the previous invoice and this invoice.

The payments are not made against this invoice.

The amount is usually displayed on this invoice, and is calculated by the billing engine.

TOTAL_ADJUSTMENTS

NUMBER (19,3)

Not null

Total adjustments made on this account between the issuing of the previous invoice and this invoice.

The adjustments are not made against this invoice.

The amount is usually displayed on this invoice, and is calculated by the billing engine.

EARLY_PAYMENT_DISCOUNT

NUMBER (19,3)

Not null

Derived

Early payment discount (if any) that applied for this invoice.

It is derived by the billing engine, and can be re-derived, based on details in the CHARGE table.

PAYMENT_DUE_DISCOUNT

NUMBER (19,3)

Not null

Derived

Pay on time discount (if any) that was applied for this invoice.

It is derived by the billing engine, and can be re-derived, based on details in the CHARGE table.

EARLY_PAYMENT_ACCOUNT_
BALANCE

NUMBER (19,3)

Derived

Balance of the account at the EARLY_PAYMENT_DATE.

It is not set by any Convergent Billing processes, but implementation-specific processing can be used to set this column if necessary to record the account balance at the early payment date. For example, this could be set by a collections or treatment module.

PAYMENT_DUE_ACCOUNT_
BALANCE

NUMBER (19,3)

Derived

Balance of the account at the PAYMENT_DUE_DATE.

It is not set by any Convergent Billing processes, but implementation-specific processing can be used to set this column if necessary to record the account balance at the date the invoice is due to be paid. For example, this could be set by a collections or treatment module.

GENERAL_1 - GENERAL_10

VARCHAR (255)

General fields 1 - 10.

Their use and meaning is controlled by entity validation on the invoice type.

They are populated as required by invoice type expressions.

Credit Status Details

Column Name

Properties

Description

TURNOVER

NUMBER (19,3)

Revenue for the last financial year for this customer node.

It is only required if REPORT_LEVEL indicates that invoices are to be produced.

TURNOVER_CURRENCY_ID

NUMBER (8)

Currency for the above turnover value.

CREDIT_LIMIT

NUMBER (19,3)

Amount of credit that a customer node can have at any one time.

It is only required if REPORT_LEVEL indicates that invoices are to be produced.

CREDIT_LIMIT_CURRENCY_ID

NUMBER (8)

CREDIT_RATING_CODE

NUMBER (4)

Indicator for how a customer is treated regarding financial accounting practices including invoicing and payment collection.

VIP customers are given the highest rating of 1.

The credit rating is defaulted by the CUSTOMER_NODE_TYPE.

CREDIT_COMMENTS

VARCHAR (255)

Comments relating to a customer's credit rating and the limit assigned, and provides reasons and justifications for the assigned values.

TAX_CLASS_CODE

NUMBER (4)

Tax class with which this customer node is associated.

NUMBER (4)

Method of payment for the customer node.

It is only required if REPORT_LEVEL indicates that invoices must be produced.

PAYMENT_LOCATION_CODE

NUMBER (4)

Location to which payments are to be made.

Examples are postal delivery, third party institution such as a bank or post office, or in person at a specific or general customer service centre.

BANK_CODE

NUMBER (3)

Code of the bank or financial institution that a customer uses for their account that relates to invoice payments (that is, for drawing of cheques and automatic debit).

BRANCH_CODE

NUMBER (5)

Code or number of the branch of the defined financial institution that the customer node uses.

BANK_ACCOUNT_NUMBER

VARCHAR (60)

Customer's account number used for automatic payments.

BANK_ACCOUNT_REF

VARCHAR (15)

Account reference number that the customer must quote with their payment at banking institutions and other third party payment collection points.

It allows payments to be associated with a single bank account. Payment entries to this account are allocated the required user access reference number.

CARD_TYPE_CODE

NUMBER (4)

Type of credit card used for payments, and based on CARD_TYPE reference code.

It is used as default information for payment.

CARD_NUMBER

VARCHAR (60)

Main or default credit card number for making payments.

CARD_EXPIRY_DATE

DATE

Date on which the main or default credit card expires.

ASSIGNED_OPERATOR_ID

NUMBER (8)

User assigned to liaise with this customer node in all matters.

SALES_CHANNEL_CODE

NUMBER (4)

Sales channel for this customer node.

COMPANY_NUMBER

VARCHAR (15)

Official company or business registration number (for example, A.C.N).

INDUSTRY_ABBREV

VARCHAR (60)

Code to identify what type of industry or activity the customer node is involved with (for example, banking, manufacturing, government).

INDUSTRY_CODE

NUMBER (4)

Code of type of industry or activity the customer node is involved with (for example, banking, manufacturing, government).

REGION_ABBREV

VARCHAR (60)

Code or map grid reference to identify the geographic region that the customer node is located in, where the region can be based on the physical address details such as suburb.

REGION_CODE

NUMBER (8)

Code or map grid reference to identify the geographic region that the customer node is located in, where the region can be based on the physical address details such as suburb.

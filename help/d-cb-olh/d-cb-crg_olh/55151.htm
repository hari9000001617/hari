FileRegisterInsert&()

Syntax

FileRegisterInsert&(
    var LastModified~,
    const FieldNames$[],
    const FieldValues?[])

 

Description

Inserts a file record in the File Processing Register.

Attributes

Application Environment

Any

Context

Any

Method

Local

 

Parameter

Description

LastModified~

Out: Date-time when the record was inserted.

FieldNames$[]

In: Names of the fields from the FILE_REGISTER_TRE_V view whose values are to be inserted.

If INSTANCE is:

  • Specified, the function converts the PARTITION_NR to the instance name using the Rating Instance value from the CustomerPartition derived attribute table.
  • Not specified and is running a:
    • Multi-instance environment, the value of INSTANCE defaults to the current Singleview instance value.
    • Single-instance environment, the value of INSTANCE is set to NULL and PARTITION_NR is set to 1.

FieldValues?[]

In: Field values in the same order that the field names are specified in FieldNames$[].

 

Returns

File ID if successful; otherwise, an error is raised.

Remarks

The function allows an external process to register a new file. Subsequent calls to the FileRegisterUpdate&() function are required to change the state, or update the audit record information.

The following table lists the default values applied to fields.

 

Field Default Values

Field Name

Value

FILE_PROCESSING_STATUS_CODE

1 (Registered)

FILE_REGISTERED_DATE

current_date()

PARTITION_NR

One of the following:

  • Default customer partition for the instance, if running in a multi-instance environment
  • 1, if running in a single-instance environment.

PHYSICAL_DIRECTORY

DIRECTORY

PHYSICAL_FILENAME

FILENAME

 

PHYSICAL_FILENAME can contain one or more of the file tags listed in the following table.

CONTROL_FILENAME:

  • Must contain one of the following tags:
    • <basename>
    • <filename>.
  • Can contain both tags or multiple instances of the same tag. The following table describes the behaviour of each tag.

 

File Tags

Tag

Behaviour

<basename>

The control file name replaces <basename> with FILENAME, but does not include the extension.

For example, if FILENAME is processing01.xml and CONTROL_FILENAME or TEMPORARY_FILENAME is <basename>.ctl, the resulting file name is processing01.ctl.

<filename>

The control file name replaces <filename> with FILENAME and includes the extension.

For example, if FILENAME is processing01.xml and CONTROL_FILENAME or TEMPORARY_FILENAME is <filename>.ctl, the resulting file name is processing01.xml.ctl.

 

  • If REGISTER_ONLY_IND_CODE is set to 0 (FALSE) , the following operations are performed:
    • If FILE_CHECK_CODE is not specified, it is set to 1 (Check if file does not exist).
    • If PHYSICAL_FILENAME is specified in the function call and FILE_CHECK_CODE is set to:
      • 1 (Check if file does not exist), FileExists&() is called to check that file path (combination of PHYSICAL_DIRECTORY and PHYSICAL_FILENAME) does not exist.
      • 2 (Check if file exists), FileExists&() is called to check that file path exists.
    • If CONTROL_FILENAME is specified and CONTROL_FILE_OPERATION_CODE is set to:
      • 1 (Check if file does not exist), FileExists&() is called to check that the control file path (combination of PHYSICAL_DIRECTORY and CONTROL_FILENAME) does not exist.
      • 2 (Check if file exists), FileExists&() is called to check that control file path exists.
      • 3 (Delete file if it exists), FileDelete&() is called to delete existing control files.
      • 5 (Verify Checksum), FileExists&() is called to check that the control file path exists, and raise an error if the path does not exist. If CHECKSUM_TYPE_CODE is specified, checks that the checksum in the control file is correct.
    • If PHYSICAL_DIRECTORY is set to a non-NULL value, FileMakeDirectory&() is called, passing the values of DIRECTORY_PERMISSION and FILE_GROUP, to ensure that the directory exists or is automatically created. The directory's permission and group are set only when the directory is created.

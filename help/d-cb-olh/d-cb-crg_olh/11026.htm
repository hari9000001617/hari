Description

Inserts the specified account details into the ACCOUNT table. A new row is also inserted in the ACCOUNT_HISTORY table.

Attributes

Application Environment

Any

Context

Any

Method

Built-in

 

Parameter

Description

StartDate~

In: Start of the date range for the account being inserted.

LastModified~

Out: Last modified date-time of the inserted account (set to the current date-time).

FieldNames$[]

In: List of fields to be inserted.

Valid field names can be found in the ACCOUNT_TRE_V view.

The order must be by field type with string fields first, followed by integer fields, real fields and finally date fields.

If the first function interface is used, values for the field names are stored in the FieldValues?[] parameter.

If the second function interface is used, values for these field names are stored in the following four type arrays:

  • StringFieldValues$[]
  • IntegerFieldValues&[]
  • RealFieldValues#[]
  • DateFieldValues~[].

Minimum set of fields required are CUSTOMER_NODE_ID and ACCOUNT_TYPE_ID. ACCOUNT_BALANCE defaults to 0 if not specified.

FieldValues?[]

In: Field values, in the same order that their names were specified in FieldNames$[].

StringFieldValues$[]

In: Values of string fields to insert.

IntegerFieldValues&[]

In: Values of integer fields to insert.

RealFieldValues#[]

In: Values of real fields to insert.

DateFieldValues~[]

In: Values of date fields to insert.

 

Returns

1 (TRUE) if successful; otherwise, 0 (FALSE) if the new account record is not created. An error is raised if one of the following conditions occur:

  • An invalid field name is passed to the function.
  • The CUSTOMER_NODE_ID and ACCOUNT_TYPE_ID (or ACCOUNT_TYPE_NAME) fields are not specified or not given a value.
  • An account with the same name exists.
  • The specified customer node does not exist.
  • The specified start date does not fall within the effective date range of the customer node record.
  • A non-insertable field is specified for an insert (based on the ACCOUNT_TRE_V view).
  • The customer node has an account of the same type, unless the account type allows multiple accounts of the same type.

Remarks

The following should be taken into account when calling the function:

  • The customer partition, and therefore, the ID of the new account, is derived from the partition of the CUSTOMER_NODE_ID passed to the function.
  • The ACCOUNT_ID of the primary account is set to the value of the PRIME_ACCOUNT_ID field (if specified), and must be within the same customer partition as the customer. If not specified, the value is derived from the customer type of the customer.
  • The ACCOUNT_NAME of the primary account is set to the value of the PRIME_ACCOUNT_NAME field (if specified). If not specified, the value is derived by the PRIME_ACCOUNT_FORMAT_EXPR of the customer type.
  • If the account balance is not specified, it is assumed to be zero. The balance date is set to the same value as the StartDate~ parameter.
  • If the account action code field is specified, the action date is set to the current date-time.
  • The credit limit amount passed in the field value is assumed to be in the correct currency (that is, it is the responsibility of the calling routine to ensure any required currency conversion has occurred).
  • If the account being inserted is associated with a customer node that exists in the CustomerPartitionException derived attribute table, an entry is inserted into the AccountPartitionException derived attribute table for the new account.
  • The effective user must have insert security group access level for the customer associated with the record manipulated by this function call.

GROUP_ACCESS_TRE_V view lists user group access level values.

After the account is successfully inserted, the biAccountPurgeLocal&() function is called to replicate the account details to separate rating instances.

Description

Updates rows of data into a global derived attribute array. The function internally calls the biDerivedAttributeArrayUpdate&() function to perform the updates.

Context

The following groups of variables are available throughout all stages of processing a global derived attribute update rows request, except where otherwise specified:

  • System client
  • Operator
  • Miscellaneous
    • evFormCode$ contains the value GDAASVR.
    • evLastRowInd& contains the value 0 during all stages of processing, except the evaluation of the global derived attribute pre-commit expressions for the last row processed.
    • evRowStatus$ contains the value Update.
  • Derived attribute.

Parameters

Parameter

Description

derivedAttributeId&

ID of the derived attribute to update table rows for.

seqNr&[]

For standard derived attributes, the sequence numbers of the rows to update.

The sequence must match the sequence of row data specified in fieldValues?[].

For extended derived attributes, this parameter is the IDs of the extended derived attribute table rows to update.

effectiveDate~

Effective date used to identify the derived attribute history record for the derived attribute table that is updated.

lastModified~

Initial value must match the last modified date-time of the derived attribute table identified by derivedAttributeID& and effectiveDate~.

On successful return, the value is the date-time of these updates.

tableName$

Name of the derived attribute table for which the rows are updated.

effectiveStartDate~

Date on which the updates first become effective.

If not defined, the current date is used.

effectiveEndDate~

Date on which the updates cease to be effective.

If not defined, the end of time value is used.

fieldValues?[]

fieldNames?[]

fieldValues?[]

Global derived attribute table row data to be updated. Accepted forms are:

  • Array of hashes of data for one or more global derived attribute rows (fieldValues?[])
  • Array of field names and an array of arrays of field values, containing data for one or more global derived attribute rows (fieldNames?[], fieldValues?[])
  • Pair of arrays of arrays of data for one or more global derived attribute rows (fieldNames?[], fieldValues?[]).

Each array or hash must contain a unique row of derived attribute table data.

If seqNr&[] is not specified, fieldValues?[] can be used to supply one of the following:

  • Sequence numbers, for standard derived attribute table rows.
  • Extended row IDs, for extended derived attribute table rows.

If the function does not find sequence numbers or extended row IDs, the supplied hash key and indexes are used to locate the rows to update.

Note:

  • Row retrieval using hash keys is available only for extended derived attribute table rows.
  • If the function must use indexes to match a row, the indexes cannot be updated in the same row update function call because row retrieval is dependent on them.

The data can be keyed according to the corresponding column name in the DERIVED_ATTRIBUTE_EVS_V view. In this case, the data passed must match the type and value of the data stored in the database.

If attribute validation is enabled, data can be keyed by interface or display label specified in the appropriate entity validation, or in core defaults.

When keyed by a label, the data must match the type and value expected as defined in the appropriate attribute type. Denormalised fields must pass the denormalised string values. Other fields must pass data of the type defined in the attribute type.

noTre&

Flag to invoke the function when in non-TRE mode.

 

Processing

Assuming that all processing stages are enabled, the following is the order of evaluation of expressions and constraints before updates to the service derived attribute rows:

  1. Bundle direct variables if the call is nested.
  2. If seqNr&[] is not specified:
    1. For standard derived attributes, derive the sequence number for each row.
    2. For extended derived attributes, derive the extended derived attribute table row ID for each row.
  3. Decrypt any encrypted values and mask any appropriate values.
  4. Apply attribute type defaults and validation.
  5. Perform attribute-based and row validations for each new row of data to be updated:
    1. Evaluate global derived attribute defaults.
    2. Check global derived attribute mandatory fields.
    3. Evaluate attribute type validation expressions for global derived attribute.
    4. Evaluate global derived attribute row expressions.
  6. Evaluate global derived attribute pre-commit expressions for each new row of data to be updated.
  7. Unmask any masked values and encrypt any variable values configured for encryption.

On performing the updates, the following sequence of actions occurs:

  1. Evaluate global derived attribute in-transaction expressions for each row of data updated.
  2. Queue and evaluate global derived attribute post-commit expressions for each row of data updated.

After update and evaluation of the post-commit expressions:

  1. Unbundle variables that were stored in step 1 before the update.

Returns

On success, the number of rows updated is returned (which should equal the length of the fieldValues?[] parameter). If any of the updates or any of the pre-update validation steps fail, an error is raised.

INV_PAYMENT_V

The INV_PAYMENT_V view defines all invoice payments known to the system. It is used by the IGP.

A single row is returned for each payment, unless the payment is rejected, in which case two rows are returned, one with an AMOUNT equal to the negative of the other value.

Column Definitions

Column Name

Properties

Description

PAYMENT_ID

NUMBER (15)

Not null

ID for a payment.

LAST_MODIFIED

DATE

Not null

Date-time when this row was last modified.

RECEIPT_NR

VARCHAR (30)

Not null

Official receipt number assigned to a payment and corresponds to accounting requirements outside the billing system, government enforced or otherwise.

It is derived by the billing system, based on the internal payment ID.

PAYMENT_DATE

DATE

Not null

Date that the payment was entered by the user.

PAYMENT_RECEIVED_DATE

DATE

Not null

Date when the payment was received by the system.

It must be less than or equal to PAYMENT_DATE.

Note:

Only the date is stored by the Convergent Billing client for PAYMENT_RECEIVED_DATE (the time is always midnight).

ATLANTA_OPERATOR_ID

NUMBER (8)

Not null

User that entered the payment.

If the payment was entered by and automatic system as part of a batch payment, this ID is that of the user who applied the batch details.

All batch payment and other automatic billing system processes operate under a common user name.

ACCOUNT_ID

NUMBER (15)

Not null

Account against which this payment is being made.

INVOICE_ID

NUMBER (15)

Not null

Invoice on which this payment is displayed.

It is not the invoice against which the payment was made.

AMOUNT

NUMBER (19,3)

Not null

Amount of the payment in the currency of the account.

It can be negative if the payment was rejected.

PAYMENT_AMOUNT

NUMBER (19,3)

Not null

Payment amount using the currency of the payment.

It is equal to AMOUNT, unless currency is different.

PAYMENT_CURRENCY_ID

NUMBER (8)

Not null

ID of the currency in which the payment is made.

PAYMENT_LOCATION_ABBREV

VARCHAR (60)

Not null

Payment location category where the payment was received (for example, customer service centre, post, bank).

PAYMENT_LOCATION_CODE

NUMBER (8)

Not null

Payment location category where the payment was received (for example, customer service centre, post, bank).

PAYMENT_TYPE_NAME

VARCHAR (40)

Not null

ID of method of the payment.

PAYMENT_TYPE_DISPLAY_NAME

VARCHAR (60)

Not null

Name of the payment method displayed.

It defaults to PAYMENT_TYPE_NAME.

Caution:

The value can be translated into a locale-specific language and therefore differ between implementations.

PAYMENT_TYPE_ID

NUMBER (4)

Not null

ID of method of the payment.

PAYMENT_STATUS_ABBREV

VARCHAR (60)

Not null

Status of this payment. Valid values are:

  • 1 (Accepted)
  • 2 (Rejected).

It defaults to Accepted.

PAYMENT_STATUS_CODE

NUMBER (4)

Not null

Indicates the status of the payment. Valid values are:

  • 0 (Entered)
  • 1 (Applied)
  • 2 (Rejected)
  • 3 (Future Dated).

Values can be specified for status when inserting a new payment. If not specified, status defaults to:

  • 1 (Applied): For new individual payments with a PAYMENT_RECEIVED_DATE <= PAYMENT_DATE and sufficient detail to apply the new payment to an invoice (for more information, refer to biPaymentInsert&() in the Configurer's Reference Guide for Singleview Convergent Billing (Vol. 2)).
  • 2 (Rejected)

    Note:

    Payments with this status cannot be modified.

  • 3 (Future Dated): For new individual or batch payments with a PAYMENT_RECEIVED_DATE > PAYMENT_DATE.
  • 0 (Entered): For new batch payments with a PAYMENT_RECEIVED_DATE <= PAYMENT_DATE.

PAYMENT_INVOICE_CODE

NUMBER (2)

Not null

Summary of the details in the PAYMENT_INVOICE table for this payment. Valid values are:

  • 0 (Invoice): The payment is against a single invoice (distributed across zero or more receivable types).
  • 1 (Split): The payment is against one or more invoices and/or a direct credit or debit to the account.
  • 2 (Unallocated): The payment is a direct credit or debit to the account (distributed across zero or more receivable types).
  • 3 (Adjustment): The adjustment is against another adjustment (for example, where a payment is allocated to a debit note).

PAYMENT_INVOICE_ID

NUMBER (15)

If PAYMENT_INVOICE_CODE is Invoice (0), this is the ID of the single invoice in the PAYMENT_INVOICE table for this payment.

ALLOCATED_ADJUSTMENT_ID

NUMBER (15)

Derived

If PAYMENT_INVOICE_CODE is Adjustment (3), this is the ID of the single adjustment in the PAYMENT_INVOICE table.

CUSTOMER_INVOICE_STR

VARCHAR (30)

ID the invoice for which this payment was made.

It has a value only if PAYMENT_INVOICE_ID is not null.

REJECTED_DATE

DATE

If a payment is rejected, this is the date of the rejection.

PAYMENT_REJECTED_ABBREV

VARCHAR (60)

If this payment is rejected, PAYMENT_REJECTED_CODE provides the reason for rejection (for example, missing signature, inadequate funds, exceeded credit limit).

The default is based on the payment type.

It must contain a value if REJECTED_DATE is not null.

PAYMENT_REJECTED_CODE

NUMBER (8)

If this payment is rejected, it provides the reason for rejection (for example, missing signature, inadequate funds, exceeded credit limit).

The default is based on the payment type.

It must contain a value if REJECTED_DATE is not null.

GENERAL_1 - GENERAL_10

VARCHAR (255)

General fields 1 - 10.

Their use is controlled by the payment type.

Typical values are for attributes such as bank, cheque number, account name, card number, card type, and expiry date.

PARTITION_NR

NUMBER (2)

Not null

Customer partition number associated with the charge generated by the payment.

Because all charges associated with an invoice are associated with the same account (and therefore customer), when customer partitioning is implemented, all charges for an invoice have the same PARTITION_NR.

It can be used as an additional selection criterion to improve access to the CHARGE table by eliminating irrelevant CHARGE table partitions.

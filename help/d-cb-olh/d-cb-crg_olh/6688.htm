Function Overloading

The Java TRE API supports calling overloaded functions using the treVariantCall() method. Static interfaces for overloaded functions can also be generated in most cases.

For more information about overloaded functions, refer to the Configuration Fundamentals Guide for Singleview Convergent Billing.

Ambiguity can arise for the following reasons:

  • The Java TRE API does not define the types of elements being passed in a list (array), which means that the API cannot differentiate between two functions with the same name and the same number of list parameters, even if the list parameters have different types. For example, the API cannot determine which of the following functions should be called:
    • Overloaded&(A&[], B#[])
    • Overloaded&(A#[], B&[])

    A static interface built for these functions does not compile because the Java compiler outputs an error message that the 'Overloaded' method is duplicated.

  • The Java TRE API supports 'lazy parameter passing' when using the treVariantCall() method by passing a map of parameter names with their values. 'Lazy parameter passing' can lead to ambiguity if not all parameters are specified in the map.

For overloaded functions with different return types, the Java TRE API adds a return type suffix after the function name when building the static interfaces.

The following table lists examples of return type suffixes.

 

Java API Return Type Suffixes

EPM Function

Java TRE API Static Interface

Overloaded?[](A$)

public List OverloadedObjectList(String A)

Overloaded?{}(A$)

public Map OverloadedObjectMap(String A)

 

The Java TRE API resolves calls to overloaded functions, where multiple functions are compatible with the passed parameters, by calling the function with the lowest interface number that is compatible.

CHARGE_INPUT_V

The CHARGE_INPUT_V view defines accessible variables in the context of Charge at billing.

Column Definitions

Column Name

Properties

Description

CHARGE_ID

NUMBER (15)

Not null

ID of a charge.

LAST_MODIFIED

DATE

Not null

Date-time when this row was last modified.

CHARGE

NUMBER (22,6)

Value of this charge.

It is mandatory, unless SUBTOTAL_DATE is not null.

PROPORTION

NUMBER (22,6)

Proportion of the charge that this charge corresponds to.

If TARIFF_ID is not null, it must contain a value.

It is verified by the DVP.

Where it contains a value, it is usually 1.

CURRENCY_ID

NUMBER (8)

ID of the currency in which this charge was made.

It is mandatory for all charges, but not required for subtotals that are aggregations of non-currency amounts.

ACCOUNT_ID

NUMBER (15)

Not null

Account to which this charge was allocated.

All charges (billable and non-billable) must be associated with an account.

The customer node is identified for the aggregation of the charges by the billing engine using the account.

UNINVOICED_IND_CODE

NUMBER (1)

Set to 1 (TRUE) if this charge is not assigned to an invoice.

All records entered by processes other than the BGP have it set to 1. The BGP clears this column after it has processed the charge and assigned it to an invoice.

It must be set to 1 if INVOICE_ID is null, and null if INVOICE_ID contains a value.

The mandatory values are enforced by the C_CHARGE check constraint.

GL_CODE_ID

NUMBER (8)

ID of the GL code associated with this charge.

Only billable charges can have a GL code associated with them.

FROM_GL_CODE_ID

NUMBER (8)

ID of an optional second GL code to be associated with this charge.

It can only be specified if this charge does not have a matching charge against another account.

It allows generation of balanced transactions to the general ledger system, without all Convergent Billing charges having to be balanced.

RECEIVABLE_TYPE_ID

NUMBER (8)

ID of the receivable type associated with this charge, if any.

For charges generated by tariffs or subtotals, it identifies the receivable type to which this charge is aggregated for this invoice.

For applied payments and adjustments, it identifies the receivable type against which a portion of the payment or adjustment was made.

NORMALISED_EVENT_ID

NUMBER (15)

Normalised event (call data) record where this charge was calculated.

It is present for all charges calculated by the rating engine, and all charges with a context of Normalised Event calculated by the billing engine.

SERVICE_ID

NUMBER (15)

Service associated with this charge (if any).

It is present for all charges calculated by the rating engine with a context of Service, and all charges calculated by the billing engine with a context of Normalised Event or Service.

CUSTOMER_NODE_ID

NUMBER (15)

Customer node associated with this charge, if any.

It is present for all charges calculated by the billing engine, irrespective of context.

Note:

It is the customer node that is the SOURCE of the charge; that is, the customer node that the billing engine was processing that causes the generation of this charge. It is not necessarily equal to the customer node associated with the ACCOUNT_ID.

It is always null for charges generated by the rating engine. The billing engine does not update CUSTOMER_NODE_ID for charges generated by the rating engine.

For billing engine processing in a context of:

  • Normalised Event or Service, it is set to the CUSTOMER_NODE that owns the service being processed.
  • Customer Node, it is the customer node being processed.
  • Customer, it is the customer being processed.

CHARGE_START_DATE

DATE

For any charges with a not null NORMALISED_EVENT_ID, it is equal to NORMALISED_EVENT.CHARGE_START_DATE.

It is verified by the DVP.

It is always populated by the rating engine, and only populated by the billing engine for charges generated in a context of Normalised Event.

It must always be used (in conjunction with NORMALISED_EVENT_ID) when performing joins between the CHARGE and NORMALISED_EVENT tables to improve query performance if table partitioning is in use.

CHARGE_DATE

DATE

Not null

Derived

Date-time used for the calculation of the charge.

For:

  • Usage-based charges (or any charges generated by the rating engine based on a normalised event), it is equal to CHARGE_START_DATE.
  • Charges generated by the billing engine, it is the issue date of the invoice that generated this charge (this is INVOICE_ID, or FROM_INVOICE_ID if it is not null).
  • Payments, it is equal to PAYMENT_RECEIVED_DATE for acceptance, and REJECTED_DATE for rejections.
  • Adjustments, it is equal to ADJUSTMENT_RECEIVED_DATE.

It is checked by the DVP.

TARIFF_ID

NUMBER (8)

Tariff, discount, or tax that generated this charge, if any.

SUBTOTAL_ID

NUMBER (8)

Subtotal, or total calculation that generated this charge.

SUBTOTAL_DATE

DATE

Date-time calculated by the given subtotal, if this subtotal generates a date result time.

It can only contain a value if SUBTOTAL_ID is not null and CHARGE is null.

It is verified by the DVP.

ADJUSTMENT_ID

NUMBER (15)

Adjustment that generated this charge.

PAYMENT_ID

NUMBER (15)

Payment that generated this charge.

INVOICE_TXT

VARCHAR (255)

Optional invoice text that can be displayed in association with this charge.

FROM_INVOICE_ID

NUMBER (15)

Invoice that resulted in the generation of this charge.

It is only set by the billing engine for charges generated by the billing engine and directed to an account outside of the current customer hierarchy being processed by the billing engine.

INVOICE_ID

NUMBER (15)

Invoice or statement with which this charge is associated.

INVOICE_SECTION_NR

NUMBER (8)

Invoice section number associated with this charge.

INVOICE_ITEM_NR

NUMBER (8)

Invoice item number.

KEY_VALUE

VARCHAR (255)

Hash key value or array index value associated with this charge for this tariff or subtotal.

It is only populated for charges generated by hash or array tariffs and subtotals.

For:

  • Hash-based tariffs or subtotals, it stores the string hash key associated with this charge.
  • Array tariffs or subtotals, it stores the integer array index associated with this charge.

GENERAL_1 - GENERAL_20

VARCHAR (255)

General fields 1 - 20.

They can be used to store intermediate calculations and lookups performed as part of tariff evaluation during rating or billing (for example, distance band, time band, multi-time band flag). The information contained in them is accessible by the BGP and the IGP.

Only charges associated with tariffs can populate these columns.

They are always null for charges associated with subtotals, payments, or adjustments.

PARTITION_NR

NUMBER (2)

Customer partition number associated with the charge, derived from information in the CustomerPartitionRange derived attribute table, using the ACCOUNT_ID column as the index for the lookup.

It forms part of the partitioning key for the table.

All charges generated before Convergent Billing v6.00 have a value of 1.

NORMALISED_EVENT_
PARTITION_NR

NUMBER (2)

Equal to PARTITION_NR of the normalised event for any charges associated with an event, and is checked by the DVP.

It is mandatory only if NORMALISED_EVENT_ID is specified.

It must be used (in conjunction with NORMALISED_EVENT_ID and CHARGE_START_DATE) when performing joins between the CHARGE and NORMALISED_EVENT tables.

It improves query performance by allowing partition elimination to be performed on the NORMALISED_EVENT table.

All charges generated before Convergent Billing v6.00 have a value of 1.

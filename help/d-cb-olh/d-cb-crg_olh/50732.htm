INV_CONTRACT_V

The INV_CONTRACT_V view defines the accessible set of columns for viewing the current contracts associated with the root customer node of the current invoice or statement.

Column Definitions

Column Name

Properties

Description

INVOICE_ID

NUMBER (15)

Not null

ID of the invoice or statement for which contracts are to be displayed.

CONTRACT_ID

NUMBER (15)

Not null

ID of a contract.

CONTRACT_NAME

VARCHAR (40)

Not null

Name for the contract.

It is the external ID for the contract, and usually automatically assigned on contract creation by the contract types entity validation.

CUSTOMER_REFERENCE

VARCHAR (40)

Customer reference for the contract.

It enables the customer to identify the contract within their own accounts payable (or equivalent) system.

Convergent Billing does not enforce uniqueness on this column.

EFFECTIVE_START_DATE

DATE

Not null

Start date-time for the effective date of this record.

The finest granularity for these date ranges is one day.

The time component always corresponds to midnight (that is, 00:00:00 hrs).

Set to StartDate~ for a new contract.

EFFECTIVE_END_DATE

DATE

Not null

End date-time for the effective date of this record.

The finest granularity for these date ranges is one day.

The time component for this column always corresponds to 23:59:59 hrs.

It defaults to the end of time for a new contract.

CONTRACT_TYPE_NAME

VARCHAR (30)

Not null

Name of contract type entered into.

It controls products that can be associated with the contract, and also defaults many characteristics on contract creation.

It cannot be modified.

CONTRACT_TYPE_DISPLAY_
NAME

VARCHAR (60)

Not null

Name of the contract type displayed.

It defaults to CONTRACT_TYPE_NAME.

Caution:

The value can be translated into a locale-specific language and therefore differ between implementations.

CONTRACT_TYPE_ID

NUMBER (8)

Not null

Type of contract entered into.

It controls the products that can be associated with the contract, and also defaults many characteristics on contract creation.

It cannot be modified.

CUSTOMER_NODE_ID

NUMBER (15)

Not null

Details of the root customer node that has entered into this contract.

It must exist as a root node for at least the effective date range of this record.

It is checked by the DVP.

It cannot be modified using the biContractUpdate&() function.

It can be modified indirectly by moving the customer to another hierarchy.

PERSON_ID

NUMBER (15)

Not null

Primary contact person for this contract.

Any correspondence that relates to this contract is addressed to this person (customer signatory or other nominated person).

It defaults to the primary contact of the root customer node.

CONTRACT_STATUS_CODE

NUMBER (8)

Not null

Status of this contract. Valid values are:

  • 1 (In Negotiation)
  • 2 (Pending)
  • 3 (Active)
  • 8 (In Grace)
  • 9 (Cancelled).

Of these values, only the statuses of Active and Cancelled have significance.

Values are client configurable. The value defaults from the CONTRACT_TYPE.

If a contract is active, all companion product instances associated with this contract must also be active for the active date range of the contract if their base product instance is active; otherwise, they must have a status of Provisioned, or the same status as their base product instance. It is checked by the DVP.

Users cannot manually modify the status of companion products associated with a contract.

If a contract is cancelled, all companion product instances associated with this contract must also have a status of Cancelled. It is checked by the DVP.

Note:

No relationship exists between contract statuses and the status of base product instances or services associated with the contract.

CONTRACT_STATUS_ABBREV

VARCHAR (60)

Not null

Status of this contract. Valid values are:

  • 1 (In Negotiation)
  • 2 (Pending)
  • 3 (Active)
  • 8 (In Grace)
  • 9 (Cancelled).

Of these values, only the statuses of Active and Cancelled have significance.

The set of possible values is client configurable. The value defaults from the CONTRACT_TYPE.

If a contract is active, all companion product instances associated with this contract must also be active for the active date range of the contract if their base product instance is active; otherwise, they must have a status of Provisioned, or the same status as their base product instance. It is checked by the DVP.

Users cannot manually modify the status of companion products associated with a contract.

If a contract is cancelled, all companion product instances associated with this contract must also have a status of Cancelled. It is checked by the DVP.

Note:

No relationship exists between contract statuses and the status of base product instances or services associated with the contract.

START_DATE

DATE

Set to the date-time that the contract first became active for this customer. However, this is not strictly enforced.

The following rules are enforced:

  • The contract must have a status of Active at this date-time. It is checked by the DVP.
  • START_DATE is mandatory if the current contract status is Active, and is enforced by the C_CONTRACT_HISTORY check constraint.

It should default or be updated if the contract status is updated to Active and this column is null.

The default value should be set to the current date-time if this is between EFFECTIVE_START_DATE and EFFECTIVE_END_DATE; otherwise, its default value should be set to EFFECTIVE_START_DATE.

CONTRACT_DURATION

NUMBER (4)

Duration of the contract. The default value is based on the contract type.

If it is specified, EXPECTED_EXPIRY_DATE is derived using the formula:

EXPECTED_EXPIRY_DATE = START_DATE + CONTRACT_DURATION

It enforced by the C_CONTRACT_HISTORY check constraint.

If EXPECTED_EXPIRY_DATE is manually modified, it is cleared.

It cannot be modified if the contract type has a FIXED_DURATION_IND_CODE of 1.

CONTRACT_DURATION_UNIT_
CODE

NUMBER (2)

Units of the contract duration. Valid values are:

  • 0 (Days)
  • 1 (Weeks)
  • 2 (Months)
  • 3 (Years).

The default is based on the contract type.

It cannot be modified if the contract type has a FIXED_DURATION_IND_CODE of 1.

EXPECTED_EXPIRY_DATE

DATE

Date-time when this contract is expected to expire.

It is derived from the contract start date and contract duration if the contract duration is specified. The derivation is enforced by the C_CONTRACT_HISTORY check constraint.

For contracts not of fixed duration, it can be manually updated.

It can contain a value even if START_DATE is null.

ACTUAL_EXPIRY_DATE

DATE

Date-time when the contract was terminated.

If it has a time of 00:00:00, the contract must have a status of Cancelled from this date-time; otherwise, the contract must have a status of Cancelled from midnight on the following day after this date-time.

It is checked by the DVP.

It can be set manually by the user, or automatically from a schedule that checks for expired contracts.

A user or the system can terminate a contract because of the failure of the customer to meet contract conditions, or changes in customer requirements, and penalties can be imposed for early contract termination.

The scheduler can set the date to a later value than the expected date if the contract type has a grace period.

COMMENTS

VARCHAR (2000)

General comments field for storing textual information about this contract, usually used only for complex or non-standard contracts.

OVERIDDEN_CONTRACTS

VARCHAR (2000)

Free form text list of contract entries that have been superseded by this contract.

RELATED_DOCUMENTS

VARCHAR (2000)

Free form text list of document entries that relate to this contract.

SALES_CHANNEL_CODE

NUMBER (8)

Sales channel from which this contract was obtained. The default is from the contract type.

RENEGOTIATE_DATE

DATE

Date-time at or after which a 'contract to be renegotiated' reminder is sent to the account manager.

It can be derived from the EXPECTED_EXPIRY_DATE and the contract type.

It can also be manually modified.

It is re-derived whenever the EXPECTED_EXPIRY_DATE is modified.

It must be null if EXPECTED_EXPIRY_DATE is null.

It is enforced by the C_CONTRACT_HISTORY check constraint.

GRACE_EXPIRY_DATE

DATE

Date-time after the EXPECTED_EXPIRY_DATE that a contract can remain in a non-cancelled status and not be renegotiated.

It can be derived from the EXPECTED_EXPIRY_DATE and the contract type.

It also can be manually modified, and is rederived whenever the EXPECTED_EXPIRY_DATE is modified.

The schedules responsible for sending reminder messages also cancel any contracts that have passed their GRACE_EXPIRY_DATE. If a contract does not have a GRACE_EXPIRY_DATE, it is cancelled as soon as its EXPECTED_EXPIRY_DATE has elapsed.

It must be null if EXPECTED_EXPIRY_DATE is null.

It is enforced by the C_CONTRACT_HISTORY check constraint.

SELECT_ALL_IND_CODE

NUMBER (1)

Not null

If set to 1 (TRUE), the contract is associated with all services in this customer hierarchy available for selection (based on base products specified for the contract type), and not associated with a non-cancelled contract.

Any new services created with compatible base products are also automatically added to this contract while its status is not cancelled.

It defaults to 0 (FALSE).

It can be modified. Changing it from TRUE to FALSE causes all services associated with the contract to remain associated with the contract.

GENERAL_1 - GENERAL_10

VARCHAR (255)

General fields 1 - 10 for entering contract related details.

The contents and validation of these fields is specified by entity validation associated with the contract type.

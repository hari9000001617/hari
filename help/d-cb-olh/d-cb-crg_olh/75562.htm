Task Variables (TASK_EVS_V)

The following table lists entity validation variables associated with the TASK_EVS_V view.

Type

evTaskBaseline
Duration&

BASELINE_
DURATION

Integer

Expected duration of the task (in seconds).

Defaults from the schedule, but can be overridden by the user or entity validation.

If the schedule type is configured appropriately, for new tasks it is derived based on the duration of the last 'n' successfully completed tasks.

The value must be greater than or equal to 1 and less than or equal to 2147483647.

evTaskBaselineEndDate~

BASELINE_END_
DATE

Date

Date-time the task is expected to complete.

The value is derived using evTaskBaselineStartDate~ + evTaskBaselineDuration&.

evTaskBaselineStart
Date~

BASELINE_
START_DATE

Date

Date-time the task is expected to start.

If a task does not have any prerequisites when created, the value is the same as evTaskScheduledStartDate~.

Otherwise, the value is based on evTaskScheduledStartDate~ and the baseline start date-time of its prerequisite tasks and their baseline durations.

It is updated automatically whenever evTaskScheduledStartDate~ is updated or relevant baseline details on prerequisite tasks are updated. However, baseline dates normally remain static for the duration of a task.

Name of the evTaskDependentResourceName$ resource key that resulted in the current task's status, if evTaskStatusReason& is set to Resource Compatibility or Resource Profile Limit, and evTaskDependentResourceName$ is populated.

If multiple resource keys resulted in the current task's status, one of the resource keys for evTaskDependentResourceName$ is arbitrarily selected.

Note:

If no value is specified in this field, evTaskDependentResourceName$ can still be populated.

Name of a resource that resulted in the current task's status, if evTaskStatusReason& is set to Resource Compatibility or Resource Profile Limit.

evTaskImmediateStart&

IMMEDIATE_
START_IND_
CODE

Integer

Specifies whether a task starts immediately. Valid values are:

  • 1: Task starts as soon as its prerequisites are satisfied (subject to resource profile and compatibility constraints), rather than waiting for evTaskScheduledStartDate~.

    A task with no prerequisites starts when the previous task in the schedule is completed.

  • 2 or 3: Task runs immediately, irrespective of the status of any prerequisite tasks, resource profile constraints, or resource compatibility constraints.

    These values can only be used when a task has evTaskStatus& of Stalled. However, when set, these values remain on the task during processing as a record that it was updated to run immediately.

    These values cannot be used for tasks with a status of Pending or Held.

evTaskInitialHeld
IndCode&

INITIAL_HELD_
IND_CODE

Integer

If set to 1, tasks are moved to Held status instead of being run. Manual intervention is required to execute the task.

Expressions on the schedule type can be used to notify users that tasks have entered a Held status.

evTaskInstanceName$

INSTANCE_NAME

String (9)

Name of the instance on which the task was run.

evTaskLastModified~

evOldTaskLastModified~

evTaskMaxDuration&

MAX_DURATION

Integer

Maximum allowed duration (in seconds) for each task for a schedule.

For each task, it defaults to the maximum duration set for the schedule type but can be overridden (for example, using an expression in the schedule type to calculate the value).

Valid values are between the value of evTaskBaselineDuration& and 2147483647 inclusive.

evTaskOperatorId&

evOldTaskOperatorId&

OPERATOR_ID

Integer

ID of the user who created or last modified the task.

evTaskPendingUpdate&

PENDING_
UPDATE_IND_
CODE

Integer

Specifies whether task updates require the regeneration of the task's resources or other attributes.

If this field is:

  • Set to 1 (TRUE), the task's resources or other attributes may be out of date.
  • Not set to 1, it must be NULL.

This option is set by the biTaskUpdate&() call and cleared by the scheduler after it updates the task and regenerates its resources.

evTaskPrerequisite
Failure&

PREREQUISITE_
FAILURE_IND_
CODE

Integer

Specifies whether the failure of a prerequisite task causes this task to be updated to a status of Failure (rather than Held).

If this field is:

  • Set to 1 (TRUE) and the failed prerequisite task is resubmitted, this task is not resubmitted.
  • Not set to 1, it must be NULL.

The value of this field defaults from the task's schedule.

evTaskPrerequisite
TaskId&

PREREQUISITE_
TASK_ID

Integer

ID of the prerequisite task.

If evTaskStatusReason& is set to Task Pre-requisite (5), this field contains the ID of a prerequisite task that results in the current task's status.

If multiple tasks result in the current task's status, one of the tasks is automatically selected to populate this field.

Count&

Integer

Number of tasks that must be scheduled in advance at any time.

evTaskProcess
Identifier&

IDENTIFIER

Integer

ID of the server process that is running this task.

This field is populated by the TLM process when it starts the task, and is used to stop tasks after a TLM failure.

This field is only populated if a task has a status of Running or Stopping.

evTaskPurgePeriod&

evOldTaskPurgePeriod&

Maximum number of days for which temporary data associated with the task is kept.

evTaskRepeatUnits&

evOldTaskRepeatUnits&

Integer

Number of the repeat type units specified in evTaskStartRepeatTypeCode& that must be allowed between tasks that correspond to this schedule.

For example, if set to 3 for a repeat type unit of months, a gap of three months is created between tasks.

It cannot be specified for a one-off task.

evTaskResubmittedTask
Id&

TASK_ID

Integer

ID of the resubmitted task.

If the task was created as part of the resubmission of a failed task, this field is set to the evTaskId& of the failed task.

If resubmission only updates the failed task, this field is still set to the ID of this task.

Date-time the task is scheduled to start.

For a newly-inserted task, the value is based on the schedule's repeat units and can be modified by the schedule's TASK_CREATION_EXPR.

This field can only be updated if the task has a status of Pending, Stalled, or Held.

evTaskStartRepeatType
Code&

evOldTaskStartRepeat
TypeCode&

START_REPEAT_
TYPE_CODE

Integer

Type of delay that must be allowed for tasks of this schedule.

It is combined with the value of evTaskRepeatUnits& to indicate how many units of the repeat type must be incorporated into the delay.

Valid values are (where n and k are derived from evTaskFirstDate~):

  • 1 (Month-Start)

    nth day-of-week of kth week from start of month

  • 2 (Month-End)

    nth day-of-week of kth week from end of month

  • 3 (Month-Day)

    nth day of the month

  • 4 (Month-End-Day)

    nth day of the month from end of month

  • 5 (Week)

    nth day of the week.

  • 6 (Day)
  • 7 (Hour)
  • 8 (Minute)
  • 9 (Second)

Values are defined in the REPEAT_TYPE reference type.

It cannot be specified for a one-off schedule.

evTaskStatus&

evOldTaskStatus&

TASK_STATUS_
CODE

Integer

Current execution status of the task. Valid values are:

  • 1 (Pending)
  • 2 (Running)
  • 3 (Success)
  • 4 (Failure)
  • 5 (Stalled)
  • 6 (Stopping)
  • 7 (Stopped).

evTaskStatusReason&

TASK_STATUS_
REASON_CODE

Integer

Provides an explanation of the current task's status.

For a task status of Stalled, valid values are:

  • 1 (Instance shut down)
  • 2 (TLM shut down)
  • 3 (Cluster task limit)
  • 4 (Instance task limit)
  • 5 (Task pre-requisite)
  • 6 (Resource compatibility)
  • 7 (Resource profile limit)
  • 13 (Schedule limit).

For a task status of Failure, valid values are:

  • 2 (TLM shut down)
  • 5 (Task pre-requisite)
  • 6 (Resource compatibility)
  • 8 (Failure exit)
  • 9 (Signal exit)
  • 10 (TLM startup)
  • 14 (TLM Internal).

For a task status of Stopping, valid values are:

  • 2 (TLM shut down)
  • 11 (User Request).

For a task status of Stopped, valid values are:

  • 5 (Task pre-requisite)
  • 11 (User Request)
  • 12 (Task overdue.

For at task status of Re-submitted, valid values are:

  • 2 (TLM shutdown)
  • 5 (Task prerequisite)
  • 6 (Resource compatibility)
  • 8 (Failure exit)
  • 9 (Signal exit)
  • 10 (TLM startup)
  • 11 (User Request)
  • 12 (Task overdue)
  • 14 (TLM Internal).

For at task status of Held, valid values are:

  • 5 (Task pre-requisite)
  • 11 (User Request).

For a description of each value, refer to the table below.

This field does not contain a value if the evTaskStatus& is Pending, Running, or Success.

The following table lists valid values for evTaskStatusReason&.

 

Valid Values for evTaskStatusReason&

Code

Value

Description

1

Instance shut down

Convergent Billing instance associated with the current task is not active.

2

TLM shutdown

TLM process associated with the task's Convergent Billing instance is not running or is in the process of shutting down.

3

Cluster task limit

Cluster-wide limit for the number of concurrent tasks of the schedule type is reached.

4

Instance task limit

Instance-specific limit for the number of concurrent tasks of the schedule type is reached.

5

Task pre-requisite

A prerequisite task condition has failed.

The task is configured to have a status of Held if the prerequisite task fails.

6

Resource compatibility

Task was forced to a status of Failure by the STD process due to a resource compatibility expression raising an exception.

7

Resource profile limit

A resource profile is in use; running the task would cause a resource to exceed its allocated usage.

8

Failure exit

Task completed with a non-zero exit code.

The exit code is displayed in the log.

9

Signal exit

Task terminated abnormally due to the receipt of a signal not generated by the TLM.

The signal received by the task is displayed in the log.

If a core file is generated by the task, it is renamed to core.<schedule type>.<process id> by the TLM and stored in the $ATA_DATA_SERVER_LOG directory for later analysis.

10

TLM startup

Task was forced to a status of 'Failure' as part of the TLM starting up on an instance.

11

User Request

A user has explicitly requested that the task be stopped.

12

Task overdue

Maximum duration of the task is exceeded.

13

Schedule limit

Another task for the schedule is running.

14

TLM Internal

The TLM process was unable to access details of the current task's schedule.

 

The following table lists valid values for evTaskTypeCode&.

 

Valid Values for evTaskTypeCode&

Code

Value

Schedule Type Description

1

Bill Cycle

Correspond to a standard batch invoice generation cycle (bill cycle).

Only schedules of this type can be referenced in places where the schedule corresponds to an invoice cycle (for example, in CUSTOMER_NODE_HISTORY and INVOICE_MESSAGE).

Schedules of this type create rows in the BILL_RUN table when their tasks are run for the first time.

2

Client Based Report (deprecated)

Invoke a client based report, implemented using Impromptu. This value is no longer supported.

3

Report

Report generated by the server and stored in an image format in the TASK_QUEUE_RESULT table.

The report can be viewed by clicking the Report button on the Task Detail form, or using the biReportImageRetrieve$() function.

4

Server Command

General purpose server commands.

The command performs some action on the server with progress and results displayed in the TASK_QUEUE_RESULT table.

5

Bill Run Operation

Perform an operation or set of operations on a bill run.

Schedules of this type are usually configured to commence immediately. The first parameter is the ID of the bill run on which the schedule operates.

6

Bill Run Customer Operation

Perform an operation or set of operations for a specific customer hierarchy on a bill run.

Schedules of this type are usually configured to commence immediately. The first parameter is the ID of the bill run on which the schedule operates. The second parameter is the ID of the root customer node on which the bill run operation is performed.

7

QA Bill Cycle

Correspond to a Quality Assurance or Temporary invoice generation cycle.

The first parameter references the main invoice generations bill cycle for which a QA run is performed.

8

Immediate Invoice Run

Perform an immediate invoice run for a specified customer hierarchy.

9

Immediate Statement Run

Perform an immediate statement (QA) run for a specified customer hierarchy.

10

Bill Run Operation (Prerequisite)

Perform a set of operations on a bill run associated with a prerequisite task.

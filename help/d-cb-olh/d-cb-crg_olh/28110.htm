Description

Inserts a new payment into the database. The function is similar to, and wraps the biPaymentInsert&() function.

Context

The following groups of variables are available throughout all stages of processing a payment insert request, except where otherwise specified:

  • System client
  • Operator
  • Miscellaneous
    • evFormCode$ contains the value PAYDSVR.
    • evLastRowInd& contains the value 0 during all stages of processing except for the evaluation of the payment pre-commit expressions.
    • evRowStatus$ contains the value Insert.
  • Payment.

FieldValues?{}

Data for the payment to be inserted.

The data can be keyed according to the corresponding column name in the PAYMENT_TRE_V view. When keyed according to database column name, the data passed must match in both type and value the data stored in the database.

If attribute validation is enabled, data can also be keyed by the display label specified in the appropriate entity validation, or Convergent Billing defaults.

When keyed by display label, the data passed must match the type and value expected as defined in the appropriate attribute type. Denormalised fields must pass the denormalised string values. Other fields must pass data of the type defined in the attribute type.

IdentifierKeys?{}

Populated on successful return with the values of the PAYMENT identifier fields.

 

Processing

Assuming that all processing stages are enabled, the following list is the order of evaluation of expressions and constraints before insertion of the new payment:

  1. Bundle direct variables if the call is nested.
  2. Apply attribute type defaults and validation.
  3. Evaluate payment defaults.
  4. Check payment mandatory fields.
  5. Evaluate payment attribute type validation expressions.
  6. Evaluate payment row expressions.
  7. Evaluate payment pre-commit expressions.
  8. Encrypt any variable values configured for encryption.

On performing the insertion, the following sequence of actions occurs:

  1. Evaluate payment in-transaction expressions.
  2. Queue and evaluate payment post-commit expressions.

After insertion and evaluation of the post-commit expressions:

  1. Unbundle variables that were stored in step 1 before insertion.

Returns

On success, the ID of the newly-inserted payment. If the insertion or any of the pre-insert validation steps fail, an error is raised.

Description

Updates contacts for a specified customer. The function calls biCustomerNodeContactUpdate&() to perform the updates.

Note:

Interface 3 can only be used to update a single contact.

Context

The following groups of variables are available throughout all stages of processing a customer contact update request, except where otherwise specified:

  • System client
  • Operator
  • Miscellaneous
    • evFormCode$ contains the value CUSTCNTSVR.
    • evLastRowInd& contains the value 0 for all stages of processing, except for the evaluation of customer contact pre-commit expressions for the last row processed.
    • evRowStatus$ contains the value Update.
  • Customer
  • Customer contact
    • The customer contact variables may only be available during the customer contact processing stages.

Parameters

Parameter

Description

customerId&

ID of the customer for which contacts are being updated.

A customer with the specified ID must exist in the database for the effective date specified; otherwise, an error is raised.

Keys?{}

Identifies the contact being modified.

Effective date used to identify the customer record to be updated.

Initial value must match the values of the customer record identified by customerId& and effectiveDate~, or the record identified by Keys?{}, if Keys?{} is specified.

Date on which the updates first become effective.

Date on which the updates cease to be effective.

If not defined, the end of time value is used.

Array of existing person entities whose contact data is to be updated.

fieldValues?[]

fieldValues?[]

Contact data to be updated. Accepted forms are:

  • Array of hashes of data for one or more contacts (fieldValues?[])
  • Array of field names and an array of arrays of field values, containing data for one or more contacts (fieldNames?[], fieldValues?[])
  • Pair of arrays of arrays of data for one or more contacts (fieldNames?[], fieldValues?[]).

The data can be keyed according to the corresponding column name in CUSTOMER_NODE_CONTACT_TRE_V view. When keyed by database column name, the data passed must match in both type and value the data in the database.

If attribute validation is enabled, data can be keyed by interface or display label, specified in the appropriate entity validation, or in Convergent Billing defaults. When keyed by a label, the data passed must match the type and value expected as defined in the appropriate attribute type.

Denormalised fields must pass the denormalised string values. Other fields must pass data of the type defined in the attribute type.

FieldValues?{}

Data to be updated for the customer contact.

It has the same characteristics as a single hash in fieldValues?[].

If an effective start date is not specified, the current date is used. If an effective end date is not specified, the end of time value is used.

IdentifierKeys?{}

Populated on successful return with the values of the CUSTOMER_CONTACT identifier fields derived from Keys?{}.

 

Processing

Assuming that all processing stages are enabled, the following is the order of evaluation of expressions and constraints for each contact before updates are performed:

  1. If Keys?{} are specified, calls the customer contact key lookup function to retrieve the customer node ID, person ID, and effective date.
  2. Bundle direct variables if the call is nested.
  3. Decrypt any encrypted values and mask any appropriate values.
  4. Apply attribute type defaults and validation.
  5. Evaluate contact defaults.
  6. Check contact mandatory fields.
  7. Evaluate contact attribute type validation expressions.
  8. Evaluate contact row expressions.
  9. Evaluate contact pre-commit expressions.

On performing the update, the following sequence of actions occurs:

  1. Evaluate customer in-transaction expressions.
  2. Queue and evaluate customer post-commit expressions.
  3. Evaluate contact in-transaction expressions for each updated contact.
  4. Queue and evaluate contact post-commit expressions for each updated contact.

After update and evaluation of the post-commit expressions:

  1. Unbundle variables that were stored in step 2 before update.

Returns

Number of contacts updated, which should equal the length of the personIds?[] parameter, if specified. If any of the updates or pre-update processing stages fail, an exception is raised.

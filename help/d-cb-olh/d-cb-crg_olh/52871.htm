Description

Inserts or updates external definitions, namespaces, and elements using XML schema files. The affected external definitions are incomplete and not validated.

Caution:

XML schema files must be validated with a third party tool before importing them using this function. The function performs basic checks for conformance to external definition specifications, but assumes that the XML is valid and correctly aligned with XML Schema 1.0.

Attributes

Application Environment

Any

Context

Any

Method

Local

 

Parameter

Description

FileName$

In: Name of the XML schema file to process.

FileNameList$[]

In: Array of XML schema file names to process.

ControlFlag&

In: Flag that sets the process behaviour if external definitions exist when processing an XML schema set. Valid values are:

  • 1 (Insert Only)

    Only new namesapces, definitions, or elements are inserted, and existing components remain untouched.

  • 2 (Update Only)

    Only matching namespaces, definitions, or elements within a definition are updated. If any component of the XML schema does not exist within an external definition, an error is raised. Elements that exist within an external definition and do not match the schema are deleted.

  • 3 (Insert and Update)

    All components of the schema are either updated or inserted. Elements that exist in Singleview, and which do not match the schema are deleted.

ActionLog$

Out: Log of actions that includes XML schema and external definition processing.

ValidateInd&

In: Validates each external definition for completeness.

If an external definition is valid, its status is set to Validated.

If validation fails, the external definition status is set to Validation Failure. A message is logged in ActionLog$, and processing continues.

 

Returns

1 if successful; otherwise, 0.

Remarks

XML schemas define only the external portion of external definitions; internal elements must be configured manually.

External definitions that are inserted or modified from an XML schema operation are always incomplete and not validated. After an external definition is created from an XML schema, it must be manually updated to link to a Singleview entity and its corresponding interface elements, and it must be validated.

When updating external definitions, if an external definition component does not match the XML schema file component, the component is removed. Singleview configuration and internal interfaces are not updated during update operations, and internal elements are not updated during update operations, and internal elements are not deleted. Internal elements have none of the following properties: element name, attribute name, or a reference to another external definition.

The XML file names cannot have UNIX shell metacharacters and cannot be a directory. If the file name does not include the full path, the function assumes that the path is relative to $ATA_DATA_SERVER_CONFIG/xsd.

Multiple file names and XML schemas are supported. XML schema files not related to other XML schema files can be provided and processed at the same time.

An XML schema file is processed only once. XML schemas defined in import statements in a schema file being processed are automatically imported and processed, even when importing multiple namespaces.

When inserting namespaces, the value for the schema attribute elementFormDefault is retrieved and the inserted namespace will either be marked as qualified or unqualified based on this value. Attributes are always unqualified.

Namespace prefixes must be unique within Singleview. Namespace prefixes are defined as follows:

  • If a namespace prefix is not defined in any XML schema files, it is automatically derived as <_tns<n>>, where <n> is a unique number defining the namespace prefix within Singleview (for example, _tns1).
  • If multiple namespace prefixes are defined for the same namespace, only one is used as the namespace prefix.
  • If the same namespace prefix is defined for multiple namespaces, the duplicated namespace prefixes are automatically derived as <_tns<n>>.
  • If a newly-introduced namespace prefix exists, it is automatically derived as _tnsY, where Y=1+MAX(_tnsX).

    For example, if the current maximum namespace prefix is _tns4, the next available namespace prefix is _tns5.

Namespace prefixes defined at a non-schema level are not used to define namespace prefixes.

Note:

The <_tns<n>> format is used only for derived namespace prefixes. Non-derived namespace prefixes must not match this format.

 

XML complex types with no elements are supported.

The conversion process converts all schema-level elements and creates like types; that is, elements at this level are converted into external definitions, although restrictions and default rules apply (see the System Configuration Guide for Singleview Convergent Billing).

This function uses the trexml Tuxedo server (see the Tuxedo Guide for Singleview) to transform XML schema into EPM data structures. During the XML schema processing, trexml checks if the XML is well-formed and highlights syntax errors. trexml does not check that the XML is valid.

Detailed logs are generated during XML schema and external definition processing. Depending on the called interface, errors are logged either in STDOUT or ActionLog$.

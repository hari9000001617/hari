Description

Deletes rows of data in a customer derived attribute array. The function internally calls the CustomerNodeDAUpdate&()function to perform the deletions.

Context

The following groups of variables are available throughout all stages of processing a customer derived attribute delete rows request, except where otherwise specified:

  • System client
  • Operator
  • Miscellaneous
    • EvFormCode$ contains the value CUSTSVR.
    • EvLastRowInd& contains the value 0 during all stages of processing, except the evaluation of the customer pre-commit expressions and the customer derived attribute pre-commit expressions for the last row processed.
    • EvRowStatus$ contains the value Delete for the customer derived attribute processing stages and the value Update for the customer processing stages.
  • Customer

    The customer variables may only be available during the customer processing stages and the evaluation of customer-derived attribute pre and post commit expressions.

  • Derived attribute

    The derived attribute variables may only be available during the customer-derived attribute processing stages.

Parameters

Parameter

Description

customerId&

ID of the customer for which derived attribute array rows is being deleted.

A customer with the specified ID must exist in the database for the effective date specified; otherwise, an error is raised.

effectiveDate~

Effective date used to identify the historical customer record that is updated.

lastModified~

Initial value of lastModifed~ must match the value in the derived attribute history record matching the specified customerId& and oldEffectiveDate~.

If not defined, the last modified value in CUSTOMER_NODE_HISTORY_V for the specified effectiveDate~ is used.

On successful return, the value is the date-time of these deletions.

tableName$

derivedAttributeId&

ID of the derived attribute array from which the rows is deleted.

seqNr&[]

Sequence numbers of rows to delete.

If any sequence number does not match a customer derived attribute array row, an error is raised.

Deletions occur in the specified order.

effectiveStartDate~

Date on which the deletions first become effective.

If not defined, the current date is used.

effectiveEndDate~

Date on which the deletions cease to be effective.

If not defined, the end of time value is used.

fieldValues?[]

fieldNames$[]

fieldValues?[]

Customer derived attribute rows to delete. Accepted forms are:

  • Array of hashes of data for one or more customer derived attribute row (fieldValues?[])
  • Array of field names and an array of arrays of field values, containing data for one or more customer derived attribute rows (fieldNames?[], fieldValues?[])
  • Pair of arrays of arrays of data for one or more customer derived attribute rows (fieldNames?[], fieldValues?[]).

Row data must contain complete row indexes for the derived attribute array. Result values must not be specified.

The derived attribute array rows to be deleted are identified by the index values specified in this parameter. If any row of data in this parameter does not match rows in the database, an error is raised.

The data may be keyed according to the corresponding column name in the CUSTOMER_NODE_DA_ARRAY table. When keyed according to the database column name, the data passed must match in both type and value the data stored in the database.

If attribute validation is enabled, data can also be keyed by interface or display label specified in the appropriate entity validation, or in Convergent Billing defaults.

When keyed by a label, data passed must match the type and value expected as defined in the appropriate attribute type. Denormalised fields must pass denormalised string values. Other fields must pass data of the type defined in the attribute type.

 

Processing

Assuming that all processing stages are enabled, the following is the order of evaluation of expressions and constraints before deletion of the customer derived attribute rows:

  1. Bundle direct variables if the call is nested.
  2. If seqNr&[] is not specified, derive the sequence number for each row.
  3. Decrypt any encrypted values and mask any appropriate values.
  4. Apply attribute type defaults and validation.
  5. Evaluate customer pre-commit expressions.
  6. Evaluate customer derived attribute pre-commit expressions for each new row of data to be deleted.

On performing the deletions, the following sequence of actions occurs:

  1. Evaluate customer in-transaction expressions.
  2. Queue and evaluate customer post-commit expressions.
  3. Evaluate customer derived attribute in-transaction expressions for each row of data deleted.
  4. Queue and evaluate customer derived attribute post-commit expressions for each row of data deleted.
  5. Unbundle variables that were stored in step 1 before deletion.

Returns

On success, the number of rows deleted is returned. If any of the deletions or pre-delete validation steps fail, an error is raised.

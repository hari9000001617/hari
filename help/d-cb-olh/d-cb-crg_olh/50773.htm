SERVICE_TRE_V

The SERVICE_TRE_V view defines the accessible set of columns for inserting and updating services.

Column Definitions

Column Name

Properties

Description

SERVICE_ID

NUMBER (15)

Not null

Read-only

ID of a service.

LAST_MODIFIED

DATE

Not null

Read-only

Date-time that the row was last modified.

EFFECTIVE_START_DATE

DATE

Not null

Date-time that the record becomes effective.

Its finest granularity is one second.

It must match the EFFECTIVE_START_DATE for the base product instance associated with the service. This is checked by the DVP.

EFFECTIVE_END_DATE

DATE

Not null

End date-time that the record ceases to be effective for the service.

It must match an EFFECTIVE_END_DATE for the base product instance associated with the service.

For a new service, it defaults to the sunset date-time. This is checked by the DVP.

OPERATOR_ID

NUMBER (8)

Not null

Read-only

User who created or last modified the service.

SERVICE_NAME

VARCHAR (64)

Not null

Name associated with the service.

Its format usually corresponds to how the service is to be displayed on customer invoices (for example, MSIDN number for a GSM service).

The name must be unique for the specified date range.

If not specified for a service insertion, it is automatically generated (if possible) based on the equipment allocated to the base product instance (if any).

NETWORK_NAME

VARCHAR (64)

Name associated with the service, in relation to the network, if this is different from SERVICE_NAME.

Its format usually corresponds to how the service is referenced by switching equipment and the user and events generated (for example, the full country number for a voice service or IMSE number for a GSM service).

If not specified for a service insertion, it is automatically generated (if possible), based on the equipment allocated to the base product instance (if any).

SERVICE_TYPE_NAME

VARCHAR (60)

Not null

Read-only

Name of service type.

SERVICE_TYPE_DISPLAY_
NAME

VARCHAR (60)

Not null

Name of service type displayed.

It defaults to SERVICE_TYPE_NAME.

Caution:

The value can be translated into a locale-specific language and therefore differ between implementations.

SERVICE_TYPE_ID

NUMBER (8)

Not null

Read-only

ID of service type.

SERVICE_SUBTYPE

VARCHAR (40)

Subtype of this service (for example, Dial-up and Dedicated subtypes within the service type of X.28).

It should be encoded using a reference type.

CUSTOMER_NODE_ID

NUMBER (15)

Not null

Read-only

ID if the customer hierarchy element or node that this service belongs to.

The customer node must exist for at least the duration of this record.

Note:

It is derived from the BASE_PRODUCT_INSTANCE_ID specified.

SERVICE_STATUS_CODE

NUMBER (8)

Not null

Status of the service. Valid values are:

  • 0 (Product Creation): A temporary status during the creation of a product. When product creation is complete this status is updated.
  • 1 (Provision Required)
  • 2 (Provisioned)
  • 3 (Active)
  • 4 (Suspended)
  • 5 (Temporary Disconnect Required)
  • 6 (Temporarily Disconnected)
  • 7 (Re-activation Required)
  • 8 (Cancellation Required)
  • 9 (Cancelled).

It cannot be specified in biServiceInsert&(), which automatically defaults to 0.

It is based on the SERVICE_STATUS reference type.

BASE_PRODUCT_INSTANCE_
ID

NUMBER (15)

Not null

Base product instance for the service.

For date ranging, the service is handled as a child of the product instance.

If the product instance has a status of Cancelled, so must the service. A cancelled service can be on a non-cancelled base product instance.

CREATION_ORDER

NUMBER (4)

Not null

Read-only

Creation order of the service for its base product instance.

SERVICE_RANGE_SEQNR

NUMBER (4)

Read-only

Sequence number of the service instance to be created for the creation order.

If not specified, it defaults to 1.

PERSON_ID

NUMBER (15)

Primary contact person for the service.

Any correspondence that relates to the service is addressed to this person.

It defaults to the primary contact for the customer node.

The person must exist for at least the duration of the record

CONTRACT_ID

NUMBER (15)

Contract under which the service is covered.

The contract must exist for at least the duration of the record.

The service must be compatible with the contract.

It is based on details in the CONTRACT_TYPE and CONTRACT_TYPE_PRODUCT tables.

Reference to the contract number of the contract stored outside of Convergent Billing.

REQUIRED_BY_DATE

DATE

Date-time that the current action as specified by SERVICE_STATUS_CODE is due to be completed.

If the SERVICE_STATUS_CODE is 1, 5, 7, or 8, it is mandatory; otherwise, it is left at its previous value.

PROVISIONED_DATE

DATE

Date-time that the service was provisioned for use by the exchange staff.

ACTIVE_DATE

DATE

Date-time that the service is billable to the customer.

It should correspond to the date-time that the customer has accepted the service (or at least been notified that the service is available).

BILLING_PRIORITY

NUMBER (4)

Priority that events on this service are processed in relation to other services on the same customer node.

It only has an effect if the bill generation server (BGP) processing this service is running with service level multi-processing enabled.

Services with a higher priority are processed before services with a lower priority on the same customer node. A lower numerical value indicates a higher priority.

If specified, the priority must have a value greater than or equal to 1.

BILLING_COMPLEXITY

NUMBER (10)

Numerical value that gives an indication of the expected time to perform a bill run on the service.

It is not automatically populated or directly used by the system.

GRADE_OF_SERVICE

VARCHAR (40)

Speed of the service.

It should be encoded using a reference type.

SERVICE_USAGE

VARCHAR (40)

Main usage type for the service where different types of usage are allowed.

It is dependent on the type of the service.

It should be encoded using a reference type.

COMMS_TYPE

VARCHAR (40)

Appropriate communications signalling type classification such as Synchronous or Asynchronous.

It is dependent on the type of the service.

It should be encoded using a reference type.

GL_CODE_ID

NUMBER (8)

GL code used for all financial transactions produced as part of the service.

It must exist for at least the duration of the record.

INDUSTRY_GENERAL_1 -
INDUSTRY_GENERAL_10

VARCHAR (255)

General fields for use as common attributes for a given type of service (for example, GSM services, irrespective of individual client customisation).

Client customisation should be performed using the GENERAL_1 to GENERAL_10 fields.

GENERAL_1 - GENERAL_10

VARCHAR (255)

General fields that can be defined in the type definition table.

PRODUCT_OFFER_ID

NUMBER (8)

ID of the product offer associated with this service.

The product offer must be:

  • Configured as service-specific.
  • Belong to the same hierarchy as the product offer assigned to the base product instance for this service.

If the service is not assigned a product offer, it must be set to NULL.

PRODUCT_OFFER_NAME

VARCHAR (60)

Name of the product offer associated with this service.

The product offer must belong to the same hierarchy as the product offer assigned to the base product instance for this service.

If the service is not assigned a product offer, it must be set to NULL.

PRODUCT_OFFER_NODE_ID

NUMBER (8)

ID of the product offer hierarchy node from which the product offer in PRODUCT_OFFER_ID was allocated.

If PRODUCT_OFFER_ID is:

  • NULL, it must be set to NULL.
  • Not NULL, it must not be set to NULL.

PRODUCT_OFFER_
ALLOCATION_DATE

DATE

Date-time at which this service was allocated the product offer specified in PRODUCT_OFFER_ID.

PREVIOUS_PRODUCT_OFFER_
ID

NUMBER (8)

Read-only

ID of the previous product offer associated with this service.

It is automatically modified when the product offer associated with a service is changed.

PREVIOUS_PRODUCT_OFFER_
NAME

VARCHAR (60)

Read-only

Name of the previous product offer associated with this service.

It is automatically modified when the product offer associated with a service is changed.

PREVIOUS_PRODUCT_OFFER_
NODE_ID

NUMBER (8)

Read-only

ID of the product offer hierarchy node from which the product offer in PREVIOUS_PRODUCT_OFFER_ID was previously allocated.

If PREVIOUS_PRODUCT_OFFER_ID is:

  • NULL, it must be set to NULL.
  • Not NULL, it must not be set to NULL.

DO_NOT_PURGE_IND_CODE

NUMBER (1)

Pseudo-column

Used for updates only

If set to 1 (TRUE), no notification is sent to the service cache for any changes to the service.

Notification is not required when updating services in bulk where it is known that the service cache will be restarted on completion of the load.

If not specified, the service cache is purged after the update is committed.

IGNORE_OFFER_END_DATE_
IND_CODE

NUMBER (1)

Pseudo-column

Used for inserts or updates only

If set to 1 (TRUE), the end of the availability period of the product offer specified by PRODUCT_OFFER_ID, PRODUCT_OFFER_NAME, or PRODUCT_OFFER_NODE_ID is ignored.

The insert or update succeeds if the start of the availability period of the product offer is no later than the EFFECTIVE_START_DATE of the service, regardless of the end of the availability period of the product offer.

NORMALISED_EVENT_V

The NORMALISED_EVENT_V view is on the normalised event table. It is used by the BGP and is optimised for this purpose.

The view also indirectly (using the VARIABLE_DEFN_HISTORY table) defines the format in which normalised events are transmitted between rating processes.

Chargeable (C) Party Details

Column Name

Properties

Description

C_PARTY_ID

VARCHAR (64)

Full national number or most precise ID for a chargeable party service.

It is usually equal to A_PARTY_ID, but is equal to B_PARTY_ID party for reverse charge and 008 calls, and some other third party for conference calls.

C_PARTY_NETWORK_ID

VARCHAR (64)

Network ID of the chargeable party (C party), if any.

For a GSM service, it corresponds to the IMSI number.

At least one of the party details (C_PARTY_ID or C_PARTY_NETWORK_ID) must be specified, although it is not enforced for performance reasons.

The rating engine performs a lookup by the C_PARTY_ID.

If it is defined, or returns no rows, a lookup by the C_PARTY_NETWORK_ID is performed. The second lookup can result in multiple matches.

C_PARTY_NAME

VARCHAR (64)

Name of the C party, if known.

For example, it can be provided by the switch for user-assisted calls.

C_PARTY_INTERNAL_ID

NUMBER (15)

ID of the chargeable party, used by the RGP to guarantee identification of the service involved.

If provided, it is used instead of the C_PARTY_ID or C_PARTY_NETWORK_ID to perform the service lookup.

Note:

External data feeds must not populate this column. If set and the EVENT_CLASS_CODE is not Usage (1), the billing engine includes any charges associated with this event in the current bill cycle even if they have a CHARGE_START_DATE after the bill cycle effective date.

C_PARTY_SERVICE_ID

NUMBER (15)

ID of the service to which the event should be charged.

If not specified, it is populated during event normalisation (by the ENM process).

If specified, it must equal C_PARTY_INTERNAL_ID; otherwise, a service name and network name lookup is performed using C_PARTY_ID and C_PARTY_NETWORK_ID.

Note:

C_PARTY_SERVICE_ID is a pseudo-column and is not stored in the database. If the value must be stored, assign the value to one of the general fields.

C_PARTY_TON_CODE

NUMBER (8)

Charge party type of number.

C_PARTY_CELL_ID

VARCHAR (20)

Charge party network cell ID (for example, mobile cell).

C_PARTY_CARRIER_CODE

NUMBER (8)

Carrier to which the call is destined.

C_PARTY_LOCATION_CODE

NUMBER (8)

Geographic location that specifies the origin of the call to the finest granularity supported by the system.

Higher forms of location such as country and continent can be derived from this value.

C_PARTY_ROUTE

VARCHAR (80)

Incoming or outgoing route of the chargeable party.

Date-time for charging purposes, usually derived from the switch date.

Date-time of the start of this event, used for events that extend over a period of time (for example, events associated with recurring charges).

PERIOD_END_DATE

DATE

Date-time of the end of this event, used for events that extend over a period of time (for example, events associated with recurring charges).

EVENT_SOURCE

VARCHAR (40)

Source of the event.

When used, it is usually the same as the EVENT_SOURCE specified in the NORMALISED_EVENT_FILE table.

EVENT_CLASS_CODE

NUMBER (2)

General classification of the event used for reporting purposes. Valid values are:

  • 1 (Usage event): The event is in relation to the usage of a service.
  • 2 (Recurring event): The event represents a rental period, typically generated by the RGP.
  • 3 (One-off or ad hoc event): These include automatically generated one-off events (generated by the RGP), and ad hoc events generated by users or other sources.

BILL_RUN_ID

NUMBER (15)

Bill run for which this event was generated, if it was generated specifically for a bill run.

It is always populated for events generated by the RGP (for rentals or adjustments).

It is used (in conjunction with ROOT_CUSTOMER_NODE_ID) to allow rapid revoking of rental events and adjustments by root customer.

If it is populated for an event, the BGP only processes this event as part of the specified bill run.

ROOT_CUSTOMER_NODE_ID

NUMBER (15)

ID of the root node of the customer hierarchy for which this event was generated.

It should only be specified if BILL_RUN_ID is specified.

It is always populated for events generated by the RGP (for rentals or adjustments), and used (in conjunction with BILL_RUN_ID) to allow rapid revoking of rental events and adjustments by root customer.

Note:

When non-rental events are generated with both BILL_RUN_ID and ROOT_CUSTOMER_NODE_ID specified, and if rental events are revoked for the given bill run and customer hierarchy, these non-rental events are deleted from the system and have to be regenerated.

EVENT_TYPE_CODE

NUMBER (8)

Type of event.

EVENT_SUB_TYPE_CODE

NUMBER (8)

Event subtype.

DURATION

NUMBER

Duration of call.

Units depend on event type.

VOLUME

NUMBER

Volume of data associated with call.

Units depend on the call type.

If populated by events generated by the RGP, its contents should be specified as volume per unit duration.

PULSES

NUMBER (8)

Number of meter pulses associated with call.

If populated by events generated by the RGP, its contents should be specified as pulses per unit duration.

CHARGE

NUMBER (19,3)

Charge as it was received from the switch.

Units are dependent on the switch.

If populated by events generated by the RGP, its contents should be specified as charge per unit duration.

CURRENCY_ID

NUMBER (8)

Currency for the above charge.

RATE_BAND

VARCHAR (40)

Time or rate band calculated for this call, usually based on its CHARGE_START_DATE.

It can be supplied by the switch, or derived during normalisation.

An encoded value can be stored.

RESERVATION

VARCHAR (40)

ID of the reservation that was made on call establishment for this event.

On rating this event, the system automatically clears the reservation (if it is specified), and updates the unbilled balance for the account with the charge calculated for this event.

Note:

It is not stored in the database. If it is required that the value be stored, it should also be assigned to one of the general fields below.

 

General Columns

Column Name

Properties

Description

GENERAL_1 - GENERAL_20

VARCHAR (4000)

General fields 1 - 20.

Their use is controlled by the normalised event type.

Typical values are for the derivation of derived normalised event attributes that need to be stored for reporting.

PARTITION_NR

NUMBER (2)

Not null

Customer partition number associated with the event.

It is populated by the ENM during normalisation.

It forms part of the partitioning key for the table.

All events before Convergent Billing v6.00 have a value of 1.

In single-instance environments, all rows have a value of 1.

In multi-instance environments, the ENM assigns a value to this column based on the SERVICE_ID to which the event was guided, and information in the CustomerPartitionRange derived attribute table.

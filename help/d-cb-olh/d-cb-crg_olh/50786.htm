PAYMENT_TRE_V

The PAYMENT_TRE_V view is used by functions defined for the biPayment service.

Properties

Description

FROM_ACCOUNT_ID

If the payment type specifies a From Account, the type of this account must match the type specified.

If the payment type specifies a currency, the currency of this account must match the currency specified.

TO_ACCOUNT_ID

NUMBER (15)

Not null

Read-only

Account against which the payment is made.

If the payment type specifies a To Account, the type of this account must match the type specified.

PAYMENT_INVOICE_CODE

VARCHAR (2)

Not null

Read-only

Summary of the details in the PAYMENT_INVOICE table for the payment. Valid values are:

  • 0 (Invoice): The payment is against a single invoice.
  • 1 (Split): The payment is against one or more invoices, adjustments and/or a direct credit or debit to the account.
  • 2 (Unallocated): The payment is a direct credit or debit to the account, or the TO_ACCOUNT_ID is not a prime account for a customer node.
  • 3 (Adjustment): The payment is against an adjustment (for example, where a payment is allocated to a debit note).

The following is permitted for calls to biPaymentInsert&():

  • A value of 0 (Invoice) can be specified if an INVOICE_ID is also specified and either of the following applies:
    • The specified invoice has no receivable type breakdown
    • AUTO_ALLOCATE_IND_CODE is specified and set to True.
  • A value of 2 (Allocated) can be specified if AUTO_ALLOCATE_IND_CODE is not specified. In this case, the payment is treated as a single credit amount to the account with no receivable type allocations.
  • A value of 3 (Adjustment) can be specified if ALLOCATED_ADJUSTMENT_ID is also specified.

It is based on the PAYMENT_INVOICE reference type.

INVOICE_ID

NUMBER (15)

Read-only

If the PAYMENT_INVOICE_CODE is 0 (Invoice), it is the ID of the single invoice in the PAYMENT_INVOICE table for the payment.

When inserting payments, INVOICE_ID can be specified if PAYMENT_INVOICE_CODE is 0.

ALLOCATED_ADJUSTMENT_
ID

NUMBER (15)

Read-only

If the PAYMENT_INVOICE_CODE is 3 (Adjustment), it is the ID of the single adjustment for this payment in the PAYMENT_INVOICE table.

It can be specified for calls to biPaymentInsert&() if PAYMENT_INVOICE_CODE is also specified with a value of 3.

Refer to PAYMENT_INVOICE_CODE for further restrictions.

CUSTOMER_NODE_ID

NUMBER (15)

Not null

Read-only

Customer node that originated the invoice payment, which should also be the node associated with the account.

It defaults to the customer node for the invoice.

PERSON_ID

NUMBER (15)

Read-only

Optional cross reference that identifies an individual making the invoice payment on the behalf of the invoiced customer node.

CURRENCY_SYMBOL

VARCHAR (5)

Not null

Currency of the payment.

When inserting payments, the currency is optional.

If not specified, the currency defaults to the currency of the payment type if specified; otherwise, the currency defaults to the currency of the payment To Account.

If the payment type specifies a currency, this must match the payment type currency.

The currency must exist from the CURRENCY_CONVERSION_DATE to the end of time.

Batch payments that specify a currency must match this column if the payment was not applied.

Updating this column can cause the total amount of the payment to change and be able to be converted to the currency specified in TOTAL_AMOUNT_CURRENCY_ID.

The currency can only be updated if the payment was not applied. Updating can cause the total amount of the payment to change.

CURRENCY_ID

NUMBER (8)

Not null

Currency of the payment.

When inserting payments, the currency is optional.

If not specified, the currency defaults to the currency of the payment type if specified; otherwise, the currency defaults to the currency of the payment To Account.

If the payment type specifies a currency, it must match the payment type currency.

The currency must exist from the CURRENCY_CONVERSION_DATE to the end of time.

Batch payments that specify a currency must match this column if the payment was not applied.

Updating this column can cause the total amount of the payment to change and be able to be converted to the currency specified in TOTAL_AMOUNT_CURRENCY_ID.

The currency can only be updated if the payment was not applied. Updating can cause the total amount of the payment to change.

CURRENCY_CONVERSION_
DATE

DATE

Not null

Date-time used when converting amounts between the payment currency and the currency used in the To or From Account.

It can only be updated if the payment was not applied. Updating can cause the total amount of the payment to change.

AMOUNT

NUMBER (19,3)

Not null

Amount of the payment in the currency of the payment.

A positive amount is a credit and a negative amount is a debit.

It can only be updated if the payment was not applied. Updating can cause the total amount of the payment to change.

FROM_AMOUNT

NUMBER (19,3)

Negative of the payment amount in the currency of the From Account.

It must have a value if FROM_ACCOUNT_ID is specified.

If not specified, it is automatically derived from AMOUNT using the CurrencyConversion#() function.

If FROM_ACCOUNT_ID is not specified, it cannot be specified.

If not specified when updating and any of CURRENCY_ID, AMOUNT, or CURRENCY_CONVERSION_DATE have changed, it is automatically recalculated.

It can only be updated if the payment was not applied. Updating can cause the total amount of the payment to change.

FROM_TOTAL_AMOUNT

NUMBER (19,3)

Read-only

Payment amount against the From Account.

FROM_TOTAL_AMOUNT equals FROM_ACCOUNT + FROM_ADJUST_AMOUNT.

TO_AMOUNT

NUMBER (19,3)

Not null

Payment amount in the currency of the To Account.

If not specified when inserting payments, it is calculated from AMOUNT using CurrencyConversion#.

If not specified when updating payments and any of CURRENCY_ID, AMOUNT, or CURRENCY_CONVERSION_DATE have changed, it is recalculated.

It can only be updated if the payment was not applied. Updating can cause the total amount of the payment to change.

TO_TOTAL_AMOUNT

NUMBER (19,3)

Not null

Read-only

Payment amount against the To Account.

TO_TOTAL_AMOUNT equals TO_ACCOUNT + TO_ADJUST_AMOUNT.

PAYMENT_LOCATION_CODE

NUMBER (4)

Not null

Read-only

Payment location where the payment was received (for example, a customer service centre, by post, or at a bank).

It is based on the PAYMENT_LOCATION reference type.

PAYMENT_TYPE_NAME

Not null

ID of the method of the payment.

Not null

The value can be translated into a locale-specific language and therefore differ between implementations.

PAYMENT_TYPE_ID

Not null

Read-only

PAYMENT_STATUS_CODE

NUMBER (4)

Not null

Indicates the status of the payment.

Values can be specified for status when inserting a new payment.

If not specified, status defaults to:

  • 1 (Applied): For new individual payments with a PAYMENT_RECEIVED_DATE <= PAYMENT_DATE and sufficient detail to apply the new payment to an invoice.
  • 2 (Rejected)

    Note:

    Payments with this status cannot be modified.

  • 3 (Future Dated): For new individual or batch payments with a PAYMENT_RECEIVED_DATE > PAYMENT_DATE.
  • 0 (Entered): For new batch payments with a PAYMENT_RECEIVED_DATE <= PAYMENT_DATE.

It is based on the PAYMENT_STATUS reference type.

AUTO_ALLOCATE_IND_CODE

NUMBER (1)

Pseudo-column

Used for inserts and updates only

Specifies if automatic allocation is performed.

If a non-zero value is specified, automatic allocation of the payment amount to invoices and receivable types is performed.

FROM_ADJUST_GL_CODE_ID

FROM_ADJUST_AMOUNT

NUMBER (19,3)

Amount that the From Account is to be adjusted to allow for discrepancies.

An amount of zero indicates a discrepancy does not exist.

If FROM_ACCOUNT_ID is specified, it defaults to the value returned from the adjust function specified by the payment type.

If the payment type does not specify an adjust function or the currency of the payment and the From Account match, it defaults to zero.

If FROM_ACCOUNT_ID is not specified, it must not be specified.

If not specified when updating payments and FROM_AMOUNT is changed, it is recalculated.

It can only be updated if the payment was not applied; updating can cause the total amount of the payment to change.

TO_ADJUST_AMOUNT

NUMBER (19,3)

Not null

Amount that the To Account is to be adjusted to allow for discrepancies.

An amount of zero indicates a discrepancy does not exist.

It defaults to the value returned from the adjust function specified by the payment type.

If the payment type does not specify an adjust function or the currency of the payment and the To Account match, it defaults to zero.

If not specified when updating payments, and TO_AMOUNT is changed, FROM_ADJUST_AMOUNT is recalculated.

It can only be updated if the payment was not applied; updating can cause the total amount of the payment to change.

Note:

It is recommended that the most recent changes be added to the start of the current text.

 

Changes to Total Amount of Payment

If an update causes a change to the currency, AMOUNT, FROM_AMOUNT, TO_AMOUNT, FROM_ADJUST_AMOUNT, or TO_ADJUST_AMOUNT, any entries in the PAYMENT_INVOICE table for the payment must be corrected (using biPaymentInvoiceUpdate&) to match the new amount.

Any entries in the PAYMENT_ITEM table must also be corrected (using biPaymentItemUpdate&) to match the new amount. If the entries in the PAYMENT_INVOICE and PAYMENT_ITEM table are not corrected, DVP errors occur.

NORMALISED_EVENT_REP_V

The NORMALISED_EVENT_REP_V view displays details on normalised events. It is used for reporting purposes. A normalised event may correspond to reconciled or unreconciled call data, and events generated by other systems. Normalised call data records are required in the database to perform traffic reporting functions.

Column Definitions

Column Name

Properties

Description

DATE

NUMBER (8)

PARENT_NORMALISED_EVENT_
ID

NUMBER (15)

For conference calls, or calls with multiple components costed separately, it is the parent event record that relates to this component.

The parent event record is assigned the cost of its components, and any direct cost associated with itself.

NORMALISED_EVENT_FILE_ID

NUMBER (15)

Not null

ID of the file from which this normalised event record was obtained.

All events are associated with a file, even if the event did not result in the creation of a file.

FILE_RECORD_NR

NUMBER (10)

Not null

Number of the record within this file, starting at 1 for the first record and incremented sequentially for each subsequent record.

Its value is assigned by the rating engine.

 

Reprocessing Details

Column Name

Properties

Description

ORIGINAL_FILE_ID

NUMBER (15)

ID of the normalised event file from which this event was originally processed.

If specified, it indicates that this event is a reprocessed event.

ORIGINAL_FILE_RECORD_NR

NUMBER (10)

Number of this record within the original file.

It must contain a value if ORIGINAL_EVENT_FILE_ID is specified.

REPROCESSED_COUNT

NUMBER (10)

Number of times this event was reprocessed.

It must contain a value if ORIGINAL_EVENT_FILE_ID is specified.

If it contains a value, the value must be greater than or equal to 1.

Chargeable (C) Party Details

Column Name

Properties

Description

C_PARTY_ID

VARCHAR (40)

Full national number or most precise ID for a chargeable party service.

It is usually equal to A_PARTY_ID, but is equal to B_PARTY_ID party for reverse charge and 008 calls, and some other third party for conference calls.

C_PARTY_NETWORK_ID

VARCHAR (40)

Network ID of the chargeable party (C party), if any.

For a GSM service, this corresponds to the IMSI number.

At least one of the party details (C_PARTY_ID or C_PARTY_NETWORK_ID) must be specified, although it is not enforced for performance reasons.

The rating engine performs a lookup by the C_PARTY_ID.

If it is defined, or returns no rows, a lookup by the C_PARTY_NETWORK_ID is performed.

The second lookup can result in multiple matches.

C_PARTY_NAME

VARCHAR (40)

Name of the C party, if known.

For example, it can be provided by the switch for user-assisted calls.

C_PARTY_INTERNAL_ID

NUMBER (15)

ID of the chargeable party, used by the RGP to guarantee identification of the service involved.

If provided, it is used instead of the C_PARTY_ID or C_PARTY_NETWORK_ID to perform the service lookup.

Note:

External data feeds must not populate this column. If it is set and the EVENT_CLASS_CODE is not Usage (1), the billing engine includes any charges associated with this event in the current bill cycle even if they have a CHARGE_START_DATE after the bill cycle effective date.

C_PARTY_TON_CODE

NUMBER (8)

Charge party type of number.

C_PARTY_CELL_ID

VARCHAR (20)

Charge party network cell ID (for example, mobile cell).

C_PARTY_CARRIER_CODE

NUMBER (8)

Carrier to which the call is destined.

C_PARTY_LOCATION_CODE

NUMBER (8)

Geographic location that specifies the origin of the call to the finest granularity supported by the system.

C_PARTY_ROUTE

VARCHAR (80)

Incoming or outgoing route of the chargeable party.

Miscellaneous Details

Column Name

Properties

Description

SWITCH_START_DATE

DATE

Date-time on CDR as produced by the network switch equipment.

CHARGE_START_DATE

DATE

Date-time for charging purposes, usually derived from the switch date.

PERIOD_START_DATE

DATE

Start date-time of the rental period covered by this event.

PERIOD_END_DATE

DATE

End date-time of the rental period covered by this event.

EVENT_SOURCE

VARCHAR (40)

Source of this event.

If used, it is usually the same as the EVENT_SOURCE specified in the NORMALISED_EVENT_FILE table.

EVENT_CLASS_CODE

NUMBER (2)

General classification of the event used for reporting purposes. Valid values are:

  • 1 (Usage event): The event is in relation to the usage of a service.
  • 2 (Recurring event): The event represents a rental period, typically generated by the RGP.
  • 3 (One-off or ad hoc event): These include automatically generated one-off events (generated by the RGP), and ad hoc events generated by users or other sources.

EVENT_TYPE_CODE

NUMBER (8)

Type of event.

EVENT_SUB_TYPE_CODE

NUMBER (8)

Event subtype.

As they are a child reference type of the event type, they are not denormalised in this view.

DURATION

NUMBER

Duration of call.

Units depend on event type.

VOLUME

NUMBER

Volume of data associated with call.

Units depend on the call type.

If populated by events generated by the RGP, its contents should be specified as volume per unit duration.

PULSES

NUMBER (8)

Number of meter pulses associated with call.

If populated by events generated by the RGP, its contents should be specified as pulses per unit duration.

CHARGE

NUMBER (22,6)

Charge as it was received from the switch.

The units are dependent on the switch.

If populated by events generated by the RGP, its contents should be specified as charge per unit duration.

RATE_BAND

VARCHAR (40)

Time or rate band calculated for this call, usually based on its CHARGE_START_DATE.

It can be supplied by the switch, or derived during normalisation.

An encoded value can be stored.

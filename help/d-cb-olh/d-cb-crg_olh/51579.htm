ACCOUNT_HISTORY_TRE_V

The ACCOUNT_HISTORY_TRE_V view provides account transaction history for each account in the system.

The following columns must not be used if the GL subsystem is enabled (in which case, other tables are used to record Sales Journal transactions that have been uploaded to the General Ledger):

  • GL_UPLOADED_IND_CODE
  • GL_UPLOADED_DATE
  • TRANSACTION_GL_FROM_DATE
  • TRANSACTION_GL_TO_DATE.

Column Definitions

Column Name

Properties

Description

ACCOUNT_ID

NUMBER (15)

Not null

ID of an account.

SEQNR

NUMBER (8)

Not null

Number that identifies the chronological sequence in which transactions are applied to the account.

LAST_MODIFIED

DATE

Not null

Date-time the record was last modified.

EFFECTIVE_START_DATE

DATE

Not null

Date-time the transaction occurred.

Multiple transactions can have the same effective start date-time.

EFFECTIVE_END_DATE

DATE

Not null

One second before the date-time the next transaction occurred on the account.

PREVIOUS_BALANCE

NUMBER (19,3)

Not null

Previous balance on the account.

If SEQNR is 1, it must be 0.

CURRENT_BALANCE

NUMBER (19,3)

Not null

Current balance of the account between the dates specified.

INVOICE_ID

NUMBER (15)

Invoice that caused the CURRENT_BALANCE to be updated for the account.

PAYMENT_ID

NUMBER (15)

Payment that caused the CURRENT_BALANCE to be updated for the account.

ADJUSTMENT_ID

NUMBER (15)

Adjustment that caused the CURRENT_BALANCE to be updated.

GL_UPLOADED_IND_CODE

NUMBER (1)

Set to 1 (TRUE) if the transaction is uploaded to the General Ledger system; otherwise, it is NULL.

Note:

Uploaded transactions cannot be modified.

GL_UPLOADED_DATE

DATE

Date-time the transaction is uploaded to the general ledger.

TRANSACTION_GL_FROM_DATE

DATE

Value for adjustment and payment transactions not uploaded to the general ledger.

TRANSACTION_GL_TO_DATE

DATE

Value for adjustment and payment transactions not uploaded to the general ledger.

The value must be greater than or equal to TRANSACTION_GL_FROM_DATE.

APPLIED_DATE

DATE

Date-time the transaction was applied or the payment rejected.

CUSTOMER_INVOICE_STR

VARCHAR (30)

ID generated for the invoice.

BILL_RUN_ID

NUMBER (15)

ID of the bill run that generated the invoice.

INVOICED_ACCOUNT_ID

NUMBER (15)

Account in which all charges associated with the invoice are held.

It can only contain a value if INVOICE_AMOUNT is 0 and STATEMENT_AMOUNT is not null.

Note:

If null and STATEMENT_AMOUNT is not null, the record is a statement for an 'Other Account'. In this case, the STATEMENT_AMOUNT is applied to the account rather than the INVOICE_AMOUNT.

CUSTOMER_ACCOUNT_INVOICE_STR

VARCHAR (30)

Optional ID generated for the invoice.

If defined, the ID is only unique within the account.

IMAGE_GENERATED_IND_CODE

NUMBER (1)

Set to True (1) when all images for the invoice have been successfully processed by the IGP, including images that have been suppressed.

It is set by the IGP.

ORIGINAL_PAYMENT_DUE_DATE

DATE

Date the invoice was originally due to be paid.

PAYMENT_DUE_DATE

DATE

Date the invoice is due to be paid.

It is usually the same as ORIGINAL_PAYMENT_DUE_DATE.

EARLY_PAYMENT_DATE

DATE

Date by which the customer must make a payment to receive an additional discount.

The discount will have been included in the INVOICE_TOTAL.

INVOICE_AMOUNT

NUMBER (19,3)

Total amount of charges associated with the invoice, excluding the balance forward.

If handling of liability accounts is:

  • Enabled, a positive value is always a debit, irrespective of the class of the account to which it is being applied.
  • Not enabled, a positive value is a debit for an asset account and a credit for a liability account.

Its values are at the limits of double precision.

STATEMENT_AMOUNT

NUMBER (19,3)

Total amount of charges associated with the statement.

It can only contain a value if INVOICE_AMOUNT is 0.

For secondary accounts, if handling of liability accounts is enabled, a positive value is always a debit, irrespective of the class of the account to which it is being applied.

BALANCE_FORWARD

NUMBER (19,3)

Balance of the account before the application of the invoice.

ACCOUNT_BALANCE

NUMBER (19,3)

Account balance after the invoice is generated.

ACCOUNT_INITIAL_DUE

NUMBER (19,3)

Initial amount due on the account.

It is usually equal to ACCOUNT_BALANCE, but can be different due to cents rounding or the presence of payment plans for the account.

CURRENT_DUE

NUMBER (19,3)

Current amount left to be paid on the invoice.

TOTAL_PAYMENTS

NUMBER (19,3)

Total payments made on the account between the issue of the previous invoice and this invoice.

TOTAL_ADJUSTMENTS

NUMBER (19,3)

Total adjustments made on the account between the issue of the previous invoice and this invoice.

EARLY_PAYMENT_DISCOUNT

NUMBER (19,3)

Early payment discount applied for the invoice (if any).

PAYMENT_DUE_DISCOUNT

NUMBER (19,3)

Pay-on-time discount applied for the invoice (if any).

LATE_PAYMENT_CHARGE

NUMBER (19,3)

Late payment charge applied to the account if payment is not made by the due date.

EARLY_PAYMENT_ACCOUNT_BALANCE

NUMBER (19,3)

Derived

Balance of the account at the EARLY_PAYMENT_DATE.

PAYMENT_DUE_ACCOUNT_BALANCE

NUMBER (19,3)

Derived

Balance of the account at the PAYMENT_DUE_DATE.

DISPUTED_AMOUNT

NUMBER (19,3)

Derived

Amount of INVOICE_AMOUNT that is in dispute.

A positive value indicates a credit to the account, irrespective of the class of the account, and the setting of the ACCOUNT_ENABLED reference type.

PAYMENT_BATCH_ID

NUMBER (15)

ID that defines a payment batch.

RECEIPT_NR

VARCHAR (30)

Official receipt number assigned to a payment.

It corresponds to accounting requirements outside the billing system (for example, government enforced).

PAYMENT_DATE

DATE

Date-time the user entered the payment.

PAYMENT_FROM_ACCOUNT_ID

NUMBER (15)

Account from which the payment is taken.

If specified, it usually defaults based on the payment type.

PAYMENT_TO_ACCOUNT_ID

NUMBER (15)

Account to which the payment is applied.

PAYMENT_INVOICE_CODE

NUMBER (2)

Summary of details in the PAYMENT_INVOICE table for the payment.

It is based on the PAYMENT_INVOICE reference type.

PAYMENT_INVOICE_ID

NUMBER (15)

Derived

If PAYMENT_INVOICE_CODE is 0, it is the ID of the single invoice in the PAYMENT_INVOICE table for the payment; otherwise, it is NULL.

PAYMENT_ALLOC_ADJUSTMENT_ID

NUMBER (15)

Derived

If PAYMENT_INVOICE_CODE is 3, it is the ID of the single adjustment in the PAYMENT_INVOICE table for the payment; otherwise, it is NULL.

PAYMENT_AMOUNT

NUMBER (19,3)

Amount of the payment in the currency of the account.

PAYMENT_CURRENCY_ID

NUMBER (8)

Currency of the payment.

PAYMENT_CURRENCY_AMOUNT

NUMBER (19,3)

Derived

Amount of the payment in the currency of the payment.

PAYMENT_CREDIT_AMOUNT

NUMBER (19,3)

Amount of the payment not allocated to any invoice.

It is always in the currency of the account.

PAYMENT_LOCATION_CODE

NUMBER (8)

Payment location category identifying how payments are received (for example, at a customer service centre, at a bank, or by post).

PAYMENT_TYPE_ID

NUMBER (8)

ID of the method of payment.

PAYMENT_STATUS_CODE

NUMBER (4)

Status of the payment. Valid values are:

  • 1 (Accepted)
  • 2 (Rejected).

REJECTED_DATE

DATE

If the payment is rejected, the date is recorded in REJECTED_DATE.

PAYMENT_REJECTED_CODE

NUMBER (8)

If the payment is rejected, it describes the reason for rejection (for example, missing signature, inadequate funds, or exceeding the credit limit).

ADJUSTMENT_NR

VARCHAR (30)

ID of this adjustment, which is used as a reference.

It is automatically generated, based on the ADJUSTMENT_FORMAT_EXPR of the adjustment type and the ADJUSTMENT_ID.

ADJUSTMENT_BATCH_ID

NUMBER (15)

ID of the adjustment batch.

ADJUSTMENT_TYPE_ID

NUMBER (8)

Standard category for the adjustment when entered.

If the adjustment is associated with a batch, the adjustment type defaults to the adjustment type of the batch.

ADJUSTMENT_DESCRIPTION

VARCHAR (255)

Description of the adjustment.

It is not displayed on the invoice, but entered by the user.

It may contain the justification for the adjustment.

If this adjustment is associated with a batch, it defaults to the description of the batch.

ADJUSTMENT_FROM_ACCOUNT_ID

NUMBER (15)

Account from which this adjustment is deducted.

If specified, it defaults based on the adjustment type.

ADJUSTMENT_TEXT

VARCHAR (80)

Text displayed on the customer invoice in the summary section, in addition to the details of the invoice adjustment.

It defaults to the text associated with the adjustment type.

ADJUSTMENT_TO_ACCOUNT_ID

NUMBER (15)

Account to which this adjustment is to apply.

If the account is associated with a batch adjustment, the currency of the account must be the same as the currency associated with the batch adjustments.

ADJUSTMENT_INVOICE_CODE

NUMBER (2)

Summary of the details in the ADJUSTMENT_INVOICE table for this adjustment. Valid values are:

  • 0 (Invoice): Adjustment is against a single invoice.
  • 1 (Split): Adjustment is against one or more invoices, adjustments, payments, direct credit, or direct debit.
  • 2 (Unallocated): Adjustment is a direct credit (amount is positive) or direct debit (amount is negative) to the account, or the TO_ACCOUNT_ID is not a prime account for a customer node.
  • 3 (Adjustment): The adjustment is against another adjustment, and is usually a debit note allocated to a credit note, or the reverse.
  • 4 (Payment): The adjustment is against a payment, and is usually a debit note allocated to a payment.

ADJUSTMENT_INVOICE_ID

NUMBER (15)

If ADJUSTMENT_INVOICE_CODE is 0, it is mandatory; otherwise, it must be NULL.

It is the ID of the single invoice in the ADJUSTMENT_INVOICE table for the adjustment.

ADJUSTMENT_ALLOC_ADJUSTMENT_
ID

NUMBER (15)

If ADJUSTMENT_INVOICE_CODE is 3, it is mandatory; otherwise, it must be NULL.

It is the ID of the single adjustment in the ADJUSTMENT_INVOICE table for the adjustment.

ADJUSTMENT_PAYMENT_ID

NUMBER (15)

If ADJUSTMENT_INVOICE_CODE is 4, it is mandatory; otherwise, it must be NULL.

It is the ID of the single payment in the ADJUSTMENT_INVOICE table for the adjustment.

ADJUSTMENT_CURRENCY_ID

NUMBER (8)

Currency of the adjustment.

ADJUSTMENT_AMOUNT

NUMBER (19,3)

Amount of the adjustment in the currency of the account.

A positive adjustment indicates a credit to the customer that must be included on the invoice.

A negative amount indicates a debit.

It includes any allowed adjustments associated with the currency conversion.

ADJUSTMENT_CURRENCY_AMOUNT

NUMBER (19,3)

Derived

Amount of the adjustment in the currency of the adjustment.

ADJUSTMENT_CREDIT_AMOUNT

NUMBER (19,3)

Amount of the adjustment not allocated to any invoice.

ADJUSTMENT_DATE

DATE

Date-time the adjustment was made.

ADJUSTMENT_ENABLE_AGING_IND

NUMBER (1)

If set to 1 (TRUE), any unallocated amount associated with the adjustment is aged according to the aging rules of the ADJUSTMENT_TO_ACCOUNT_ID's account type.

It can only be set to 1 if ADJUSTMENT_FROM_ACCOUNT_ID is NULL; that is, aging cannot be enabled for transfer adjustments. If not set to 1, it must be NULL.

It is usually only set for adjustments representing debit notes.

ADJUSTMENT_DUE_DATE

DATE

Date-time at which the adjustment was due.

It only has a value if ENABLE_AGING_IND_CODE is set to 1.

It is usually only set for adjustments representing debit notes.

GENERAL_1 - GENERAL_10

VARCHAR (255)

General fields 1 - 10 controlled by the payment type or adjustment type (depending on whether the record is a payment or adjustment).

Typical values are items such as bank, cheque number, account name, card number, card type, or expiry date.

Description

Updates the specified contract record.

Attributes

Application Environment

Any

Any

Method

Remote

 

Parameter

Description

ContractId&

In: ID of the contract to be updated.

OldEffectiveDate~

In: Effective date-time that identifies the contract to be updated.

LastModified~

In: Last modified date-time of the contract to be updated.

Out: Last modified date-time of the updated contract.

FieldNames$[]

In: List of fields to be updated.

String fields should be specified first, followed by integer, real, and date fields.

StringFieldValues$[]

In: Values of string fields to update.

IntegerFieldValues&[]

In: Values of real fields to update.

DateFieldValues~[]

In: Values of date fields to update.

FieldValues?[]

In: List of updated field values.

 

Returns

1 (TRUE) if successful; otherwise, an error is raised.

Remarks

The LastModified~ parameter must be set to the LAST_MODIFIED date-time of the record to be updated. A list of valid field names to be updated are passed in the array FieldNames$[] with corresponding values passed in the appropriate field type arrays.

In interface 2, the values corresponding to the field names are passed in the FieldValues?[] array.

Activating a Contract

When a contract's status is updated to Active for the first time, the status of all companion product instances associated with the contract must also be updated to Active.

Cancelling a Contract

  • If a contract's status is updated to Cancelled, the status of all companion product instances associated with the contract must also be updated to Cancelled.
  • When cancelling a contract, the ACTUAL_EXPIRY_DATE of the history record, and the new history record, is set to EFFECTIVE_START_DATE of the new record.
  • Updating a contract record's status to Cancelled always applies to the end of time as a contract cannot be updated after it is cancelled.

Contract Names

Contract names must be unique within overlapping date-time ranges.

Service Selection Flag

  • If a contract is updated with SELECT_ALL_IND_CODE = 1, it is necessary to search for all services, under the hierarchy existing for any part of the date range of the new contract, that:
    • Do not have a status of Cancelled, Moved, or Upgraded.
    • Are compatible with the contract.

    Services without a status of Cancelled, Moved, or Upgraded need to be associated with the contract using biServiceUpdate&().

  • For a given CUSTOMER_NODE_ID, no two contracts can have SELECT_ALL_IND_CODE set to a non-zero value (TRUE) for an overlapping date range if their contract types have any products in common.

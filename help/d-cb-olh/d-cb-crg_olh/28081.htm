Description

Inserts contacts for a specified customer. The function calls biCustomerNodeContactUpdate&() to perform the insertions.

Note:

Interface 3 can only be used to insert a single contact.

Context

The following groups of variables are available throughout all stages of processing a customer contact insert request, except where otherwise specified:

  • System client
  • Operator
  • Miscellaneous
    • evFormCode$ contains the value CUSTCNTSVR.
    • evLastRowInd& contains the value 0 for all stages of processing, except for the evaluation of contact pre-commit expressions for the last row processed.
    • evRowStatus$ contains the value Insert for all stages of processing, except for customer post-commit processing, where it contains the value Update.
  • Customer
  • Customer contact

    The customer contact variables may only be available during the customer contact processing stages.

Parameters

Parameter

Description

customerId&

ID of the customer for which contacts are being inserted.

A customer with the specified ID must exist for the effective date specified; otherwise, an error is raised.

effectiveDate~

Effective date used to identify the customer record to be updated.

lastModified~

LastModified~

Initial value must match the value of the customer record specified by customerId& and effectiveDate~, or FieldValues?{}.

If FieldValues?{} is specified, lastModified~ can be undefined and is automatically derived from CUSTOMER_NODE_HISTORY_V using effectiveDate~.

On successful return, set to the date-time of the insertions.

effectiveStartDate~

StartDate~

Date on which the insertions first become effective.

If not defined, the current date is used.

effectiveEndDate~

Date on which the insertions cease to be effective.

If not defined, the end of time value is used.

personIds&[]

Array of existing person entity IDs to be inserted as contacts.

fieldValues?[]

fieldNames?[]

fieldValues?[]

Contact data to be inserted. Accepted forms are:

  • Array of hashes of data for one or more contacts (fieldValues?[])
  • Array of field names and an array of arrays of field values, containing data for one or more contacts (fieldNames?[], fieldValues?[])
  • Pair of arrays of arrays of data for one or more contacts (fieldNames?[], fieldValues?[]).

The data can be keyed according to the corresponding column name in CUSTOMER_NODE_CONTACT_TRE_V view. When keyed according to database column name, the data passed must match in both type and value the data stored in the database.

If attribute validation is enabled, data can be keyed by interface or display label specified in the appropriate entity validation, or in core defaults. When keyed by a label, the data must match the type and value expected as defined in the appropriate attribute type.

Denormalised fields must pass the denormalised string values. Other fields must pass data of the type defined in the attribute type.

FieldValues?{}

Data to be inserted for the customer contact.

It has the same characteristics as a single hash in fieldValues?[].

It specifies values for the primary IDs of the customer contact. If the customer and person cannot be identified by FieldValues?{}, an error is raised.

If an effective end date is not specified, the end of time value is used.

IdentifierKeys?{}

Populated on successful return with the values of the CUSTOMER_CONTACT identifier fields.

 

Processing

Assuming that all processing stages are enabled, the following is the order of evaluation of expressions and constraints for each contact before insertions are performed:

  1. If FieldValues?{} is specified:
    • The values specified are used to retrieve the customer node ID and the person ID.
    • StartDate~ is the effective date to identify the customer historical record to be updated.
  2. Bundle direct variables if the call is nested.
  3. Decrypt any encrypted values and mask any appropriate values.
  4. Apply attribute type defaults and validation.
  5. Evaluate customer contact defaults.
  6. Check contact mandatory fields.
  7. Evaluate contact attribute type validation expressions.
  8. Evaluate contact row expressions.
  9. Evaluate contact pre-commit expressions.

On performing the insertion, the following sequence of actions occurs:

  1. Evaluate customer in-transaction expressions.
  2. Queue and evaluate customer post-commit expressions.
  3. Evaluate contact in-transaction expressions for each contact inserted.
  4. Queue and evaluate contact post-commit expressions for each contact inserted.

After insertion and evaluation of the post commit expressions:

  1. Unbundle variables that were stored in step 2 before insertion.

Returns

Number of contacts inserted, which should equal the length of the fieldValues?[] parameter if specified. If any of the insertions or pre-insert processing stages fail, an error is raised.
